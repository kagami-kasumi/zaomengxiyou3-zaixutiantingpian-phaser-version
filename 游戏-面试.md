## 30 秒概述

我在把一款十多年前的 Flash 双人动作 RPG《再续天庭》用 Phaser 3 和 TypeScript 现代化重写。原始资料包含约 20 万行反编译 AS3 和 70 多个 SWF，难点不只是迁移技术栈，还包括从时间轴、嵌套坐标和共享运行时中恢复玩法事实。

我完成并持续重做了战斗、关卡、HUD、六槽存档、天庭地图、双玩家成长数据和炼丹炉等核心系统。目前 Stage 1 三关和正式“启动 → 存档 → 地图 → 战斗 → 结算 → 解锁 → 重载”主循环已经闭合，Stage 2-1 已完成首个可玩实现并进入运行校准。

我的核心工作不是逐行翻译 AS3，而是把不完整的遗留资料转成可追溯的行为合同，再用现代架构、自动测试和运行时验收实现它。

## 项目背景与我的职责

- **项目目标**：尽量还原原版的外观、玩法、数值、手感和流程。
- **技术路线**：Phaser 3 + TypeScript，使用现代模块边界重写，不维护 Flash 工程。
- **遗留资料**：反编译 AS3、恢复 SWF、SymbolClass、MovieClip 时间轴、嵌套矩阵和旧提取结果。
- **产品特征**：原版支持本地双人，输入、HUD、库存、技能、宠物、装备、事务和存档都必须区分 owner。
- **我的职责**：负责玩法与视觉逆向、现代架构、系统实现、自动验证和运行时校准。

项目的核心难点是：

1. 反编译源码提供的是线索，不自动等于事实。
2. Flash 的类和时间轴结构不能直接成为现代架构。
3. 本地双人会放大数据所有权和生命周期问题。
4. 业务可用、资源接入和视觉复现是不同完成等级。

## 四个关键问题与解决方案

### 1. 反编译代码能读，但不能直接翻译

一个关卡行为通常不只存在于关卡类。局部 AS3 可能定义刷怪点，真正的输入、物理、镜头、死亡、结果和存档逻辑散落在共享运行时。Flash 中的 `x/y` 还可能是 MovieClip 注册点、父容器局部坐标或矩阵变换后的舞台坐标。

我使用六段证据链：

```text
局部类/时间轴
  → 共享运行时调用链
  → SWF 几何与坐标语义
  → 与架构无关的可观察行为合同
  → 现代 owner 映射
  → 确定性测试 + 运行时观察
```

关键结论分为“确认事实、交叉确认、推断、未知、现代设计选择”。证据不足时保留未知，不用方便的常量补成原版事实。

Stage 2-1 的墙体、单向平台、停点、25 个刷怪点、五批 53 怪、38 个冰刺和结果流程都先形成行为合同，再进入实现。怪物和弹体真视觉尚未闭合，因此明确使用现代占位，没有冒充原版资源。

### 2. Flash 架构不能成为现代架构模板

原版大量依赖全局对象、字符串状态、时间轴帧、显示对象副作用和跨类直接读写。逐类翻译虽然短期快，但会把 Flash 时代的耦合完整复制过来。

现代实现按职责拆分：

- Scene 负责 Phaser 对象创建、生命周期和调度。
- 纯 System 持有关卡推进、战斗、存档、库存和装备事务规则。
- Bridge 把运行时对象转换为系统快照或命令。
- Registry 和配置表提供数值、配方和资源的单一数据源。
- 持久状态与冷却、Buff 等瞬时状态分开。

战斗重做时，我没有让 Stage 1 三关继续各自维护怪物重力、伤害和奖励，而是建立共享怪物物理与死亡奖励 owner；飞行怪使用显式例外，经验、生命、魔法和灵魂奖励从统一入口结算。

### 3. 本地双人会放大所有数据边界问题

很多单机 UI 默认只有一个“当前玩家”。支持本地双人后，库存、装备、技能、宠物、暂存材料、资源消耗和存档写回都可能串号。仅在界面上增加 P1/P2 按钮并不能建立数据所有权。

我把 owner 贯穿到：

- 输入绑定、技能槽和 HUD；
- 库存、装备、法宝、宠物和成长模型；
- 合成、强化、分解和制作事务；
- 当前存档槽的 V4 持久化结构。

切换 owner、关闭页面或场景 shutdown 时，未提交材料必须退还原 owner；成功事务只写入对应玩家和当前存档槽。强化与分解使用原子提交和可注入随机，预览阶段不直接修改正式数据。

炼丹炉 112 个配方最终都在 P1/P2 两套独立库存上执行，共形成 224 条完整事务矩阵。

### 4. “资源接入”不等于“UI 复现”

早期实现曾把整页原图当背景，再覆盖现代标题、页签和按钮。业务可用，但玩家看到的是现代替代层，不是原版显示层级和交互状态。

UI 逆向因此增加显示列表与逐状态门禁：

- child、depth、父子关系、矩阵、注册点和可见边界；
- 静态/动态文字和数据来源；
- normal、hover、pressed、selected 与 hitTest；
- AS3 动态创建的列表、图标、分页和返回控件；
- 原显示对象与现代资源的一一映射；
- 原版视觉基准、现代例外和逐状态差异。

近期炼丹炉 UI 重做删除了现代覆盖导航，恢复原图承载四个操作入口，通过透明命中区连接业务，并重新校准四个业务主体、返回入口和 P1/P2 状态。

## 代表性系统重做

### 战斗、物理与奖励统一

- 三个 Stage 1 关卡共享角色移动、攻击窗口、受击保护和死亡结果。
- 地面怪默认进入共享重力系统，飞行怪使用显式 `flying` 例外。
- 怪物死亡统一进入奖励 owner，结算生命、魔法、灵魂、战意副收益和直接经验。
- `HeroProgressionModel` 和 `HeroSkillModel` 成为 HUD 的只读来源。
- 通过死亡原因分类和无调试通关基线校准难度，不凭感觉改常量。

这次重做解决的是“同一战斗规则随关卡漂移”的系统性问题。

### 正式游戏主循环

项目从早期测试场景升级为正式玩家旅程：

```text
启动
→ 新建/读取六槽存档
→ 天庭地图
→ 选择 1P/2P 与关卡
→ 战斗 HUD
→ 失败/胜利/退出
→ 解锁下一关并保存
→ 返回地图
→ 再次读档恢复
```

存档支持当前槽、六槽隔离、旧版本迁移、旧单槽导入、损坏拒读和显式删除。天庭地图集中管理节点的锁定、当前、已通关和未接入状态，不在 Scene 中复制解锁规则。

### 炼丹炉与装备事务

- **合成**：112 个权威配方、无序三槽匹配、特殊属性继承和 201 个真图标。
- **强化**：强化石概率、幸运符、灵魂成本、失败降级、神恩保底和实例属性更新。
- **分解**：可注入随机、六结果槽和成功/失败原子提交。
- **制作**：78 本可达制作书、材料、灵魂、三颗可选宝石和装备实例持久化。
- **法宝**：装备门禁、等级、成长率、五行、材料阶段、强化与重置确认。

UI 负责选择、预览和反馈，系统负责准入、消耗、随机、产出和持久化。

### UI 原生化重做

- 正式启动页、六槽存档、天庭地图和核心 HUD 使用恢复 SWF 中的真资源。
- 工坊四操作恢复原图入口、原生返回和透明命中。
- P1/P2 owner 样式、四个业务主体位置和页面生命周期重新统一。
- 技能四页已经具备正式业务和存档能力；后续继续补齐原生显示列表与逐状态视觉证据。

## 现代架构与验证链

```text
恢复 SWF / AS3 / 运行观察
            │
            ▼
证据矩阵与行为合同
            │
            ▼
Registry / Pure System / State Model
            │
            ▼
Scene + Bridge + Phaser Runtime
            │
            ├── 确定性专项测试
            ├── 系统回归与 TypeScript 构建
            └── 940×590 运行时/视觉验收
```

这条链路分开了三类事实：

- 原版事实：由 AS3、SWF 或运行观察支持。
- 现代设计：由可维护性、平台限制和用户体验决定。
- 完成证据：由测试、构建和运行验收共同提供。

## 不应过度声称的部分

- Stage 2-1 是“首个可玩实现完成、待运行校准”，不是整条功能线已经关闭。
- 怪物和部分弹体仍使用现代占位，不能声称视觉已经完整复现。
- 真资源接入不等于显示列表、字体、按钮状态和动态布局逐像素一致。
- 自动测试不能替代持键手感、镜头、时序和视觉验收。
- 现代实现追求可观察体验接近原版，不追求逐行或逐类复刻 AS3。

## 2 分钟表达示例

我在做一个老 Flash 双人动作 RPG 的现代化重写，技术栈是 Phaser 3 和 TypeScript。资料规模大约是 20 万行反编译 AS3 和 70 多个 SWF。真正的难点不是把语法换成 TypeScript，而是从时间轴、共享运行时和嵌套矩阵中恢复玩家真正看到的行为，同时不能把 Flash 的全局状态和显示对象耦合照搬过来。

我的做法是先建立六段证据链：从关卡或对象局部代码追到共享输入、物理、镜头和存档，再补 SWF 注册点、矩阵和可见边界，最终形成与架构无关的行为合同。现代实现使用纯 System、状态模型、Registry、Scene 和 Bridge 分层，最后同时做确定性测试和 940×590 运行验收。

近期我重点重做了战斗、正式主循环和炼丹炉。战斗方面把三个关卡各自漂移的重力、伤害和奖励统一到共享 owner；正式流程已经覆盖六槽存档、天庭地图、双玩家 HUD、关卡结算、解锁和重载；炼丹炉覆盖 112 个权威配方和 201 个真图标，并在 P1/P2 两套库存上跑了 224 条事务。强化、分解、制作和法宝使用原子事务与持久化，不让 UI 直接修改正式数据。

这个项目让我形成的核心方法是：遗留游戏现代化不是翻译代码，而是恢复事实、重建边界，并为行为、数据和视觉分别设计验证证据。

## 常见追问

### Q：为什么不直接把 AS3 类逐个翻译成 TypeScript？

AS3 类结构混合了玩法、Flash 时间轴、显示对象生命周期和全局状态。逐类翻译会保留错误边界。我只把可观察行为、数值和流程当合同，现代代码按 Scene、System、Bridge、Registry 和状态模型重新分配职责。

### Q：怎么证明现代实现接近原版？

行为层追踪局部代码和共享调用链，空间层核对 SWF 注册点、矩阵和可见边界，自动测试验证确定性合同，运行验收验证视觉、持键、镜头、时序和组合路径。证据不足的部分明确标记为推断、未知或现代设计选择。

### Q：为什么双玩家需要贯穿整个模型？

owner 不是 UI 筛选条件，而是数据所有权。库存、技能、宠物、装备、暂存材料、资源消耗和存档写回任何一层缺少 owner 都会串号。把 owner 从输入贯穿到事务和 V4 存档后，隔离才可测试。

### Q：如何设计炼丹炉的原子事务？

选择材料时只进入会话暂存，不直接改正式库存；系统先验证配方、准入、资源和随机条件，再一次性提交消耗与产出。切换 owner、关闭或 shutdown 会把未提交内容退回原 owner。随机源可注入，因此成功、失败和保底路径可以稳定测试。

### Q：为什么整页真背景不能证明 UI 已复现？

原版交互可能由嵌套 child、动态文字、按钮四态、透明 hitTest 和 AS3 动态列表组成。整页图只能证明底图存在。显示列表、动态数据来源、逐状态交互和原版视觉基准都闭合后，才适合声称 UI 原生化。

### Q：项目中最有代表性的系统性问题是什么？

怪物物理和死亡奖励曾随关卡漂移：某关有重力，某关固定悬空，奖励也只在部分关卡接入。修复不是改几个常量，而是建立共享物理和奖励 owner、定义飞行例外、补三关接线和测试，并让后续关卡复用这条边界。

### Q：最大的教训是什么？

第一，反编译代码提供的是线索，不自动提供事实。第二，能运行不等于已复现，尤其是 UI 和空间行为。第三，系统重做要消除规则漂移，而不只是把旧代码换一个文件存放。
