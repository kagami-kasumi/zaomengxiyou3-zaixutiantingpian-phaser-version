# 面试准备：Harness Engineering 实践经验

## 一句话概括

为一个 Flash 游戏现代化重写项目，从零设计并持续迭代了一套文档驱动的 AI Agent 协作脚手架（Harness），通过任务分级路由、Feedforward/Feedback 双层约束、代码质量门禁和轻量 DDD 统一语言，让多个 AI agent 可以在无人工交接的情况下冷启动、正确执行并收敛上下文。

---

## 项目背景

- **项目**：将 10+ 年前的 Flash 游戏《再续天庭》用 Phaser 3 + TypeScript 重写
- **规模**：约 20 万行 AS3 反编译源码，70+ SWF 资源文件，涉及双人本地合作、5 个角色、装备/背包/法宝等系统
- **约束**：多人轮换开发，每次对话由不同的 AI agent 接手，没有人工交接环节
- **我的角色**：设计和持续维护整个 AI 协作脚手架（Harness），让 agent 可以冷启动接手任何任务并正确执行

---

## 我解决的问题

### 问题 1：Agent 跨对话失忆——冷启动怎么办

AI agent 每次新对话都是空白状态，不记得上一轮做了什么、做到哪一步、为什么做某个决策。

**方案——分层文档驱动的跨对话记忆系统**：

核心思路是把项目知识拆分为职责正交的文档层，每层只回答一类问题，层与层之间通过 ID 引用（`M-*` 机制 ID、`VS-*` 切片 ID、`TASK-*` 任务 ID）形成可校验的链接网络：

| 层级 | 文档 | 回答的问题 |
|------|------|-----------|
| 战略导航 | `TASK_OUTLINE.md` | 项目去哪、怎么分阶段 |
| 任务编排 | `task-board.md` / `task-history.md` | 当前有什么活、做完了什么 |
| 逆向索引 | `mechanics-index.md` | 每个游戏机制扒清楚了吗、复现了吗 |
| 切片队列 | `vertical-slices.md` | 每个可玩切片依赖哪些前置机制 |
| 领域语言 | `glossary.md` | 同一个概念在代码里该叫什么 |
| 脚手架 | `docs/workflow/` | agent 的执行协议、质量门禁、治理日志 |

关键设计决策：

- **任务分级路由而非全量预加载**：最初的设计让 agent 每次启动读 6 个文档。后来发现这对轻量请求（解释问题、改个常量）太重了——上下文是有限资源。改造为三档分级：轻量请求只读 `AGENTS.md` + `TASK_OUTLINE.md`；正式游戏 task 按阅读分流表补读对应文档；脚手架维护额外读 workflow 目录。
- **看板/历史分离**：未完成任务和已完成任务分两个文件。新对话默认不读历史，只有追溯历史决策时才按需加载。这避免了上下文膨胀，符合 Progressive Disclosure 原则。
- **文档即 API**：文档之间通过规范化的 ID 互引用（如 task 的"目标机制/切片"列必须包含 `M-*` 或 `VS-*`），形成可机器校验的链接网络。

**效果**：不同 agent 在不同时间接手任务，读完分级的必读文档即可独立执行，不需要人工转述上下文。

### 问题 2：Agent 行为不可控——怎么让它既自由又不越界

Agent 容易顺手重构无关系统、扩大任务范围、照搬原版代码的糟糕结构、在一次对话里揽太多活。

**方案——Feedforward + Feedback 双层约束体系**：

**第一层：Feedforward（前置引导）**

- `AGENTS.md` 中的硬性禁止项（不改 `extracted_flash/`、不把方向键分配给玩家 1、不照搬 AS3 全局状态模式、不一次性重构大量系统），每条都可以被验证是否违反
- 标准化任务模板——9 个必填字段：任务类型、目标机制/切片、输入资料、输出产物、完成定义、验收标准、禁止范围、状态更新、推荐后续任务。有了"禁止范围"，agent 就知道边界在哪
- 冷启动阅读分流表——按任务类型（轻量请求 / 游戏执行 / 玩法逆向 / 代码实现 / 领域命名 / 任务生成 / 脚手架维护 / 历史追溯）给出最小必读文档集
- 代码质量门禁——`src/scenes/` 只能做 Phaser 对象创建和系统调度，复杂战斗/关卡/技能/背包/存档/AI 规则必须进 `src/systems/`；每个可受击/可交互实体必须有稳定 ID

**第二层：Feedback（后置校验）**

- `validate-workflow.mjs`（约 480 行）自动检测 15+ 条一致性规则：必读文件是否存在、任务行与定义是否双向匹配、Done 任务是否已迁移至历史、推荐任务是否存在且 Ready、`M-*`/`VS-*` 引用是否有效、Ready 任务的依赖机制是否已扒清、治理日志是否覆盖最新文件变更、`package.json` 的必要 scripts 是否存在、CLAUDE.md 是否指向代码质量门禁、领域词汇表结构完整性、`src/` 中是否出现禁止别名……
- 区分 error 和 warning：破坏性不一致报 error（exit 1），可容忍的不完美报 warning（不阻断 CI）
- `npm run check:workflow` → `npm run test:systems` → `npm run build` 三级验证链，加总为 `npm run check:all`

**第三层：对话收束规则**

这是从反复踩坑中提炼出来的。agent 有个坏习惯——做了少量工作就建议"新开个对话继续"。于是明确规则：同一个正式 task 未完成时默认继续当前对话；上下文过长优先 compact，compact 后复查关键文件；只有 task 完成、切换明显不同子系统，或已读取大量 AS3/逆向/历史资料时，才建议新开对话。同时明确"不要因为只做了少量工作就建议切换"——这条规则是事后从事故事件中总结的，不是事前推演的。

**效果**：校验脚本从约 200 行演进到约 480 行，覆盖范围从原来的 10+ 条规则扩展到 15+ 个校验维度。规范从"靠人遵守"变成"跑不过就报错"。

### 问题 3：逆向工程和实现之间的衔接——扒代码和写代码怎么不脱节

需要先逆向理解原版 AS3 代码的玩法逻辑，再决定怎么用现代方式实现。两者的依赖关系如果不显式化，agent 容易边猜边写。

**方案——双状态跟踪 + 纵向切片驱动**：

- `mechanics-index.md` 对每个机制同时维护两个状态：**逆向状态**（未扒 / 部分已扒 / 已扒 / 暂缓）和**复现状态**（未复现 / 部分复现 / 已复现）。一个机制可能"扒清楚了但还没实现"，也可能"实现了一部分但还有些细节没扒清"——两种状态独立更新。
- `vertical-slices.md` 按可试玩切片组织实现队列，每个切片明确列出依赖的前置机制（用 `M-*` ID），并标记当前切片整体的实现状态。
- 校验脚本自动检查：如果一个 task 状态为 Ready，但它依赖的机制逆向状态仍为"未扒"，直接报 error；如果依赖的机制仅为"部分已扒"，报 warning 提醒 agent 注意。
- 任务类型严格分家：`TASK-SETTINGS`（只读不写代码）→ `TASK-ARCH`（基础设施）→ `TASK-SLICE`（可玩闭环），确保每个任务职责单一。

**效果**：agent 执行 TASK-SLICE 前自动检查依赖机制的逆向状态，避免"不知其所以然"的实现。

```
        │                                               │
        └───────────────────┬───────────────────────────┘
                            ▼
                    ┌───────────────┐
                    │   代码层       │
                    │ Phaser 3 + TS │
                    │ Scene/System/ │
                    │ Core 分层     │
                    └───────────────┘
```

核心设计原则：

- **职责正交**：每个文档只回答一类问题，不重复。改路线找 `TASK_OUTLINE`，查任务找 `task-board`，看机制找 `mechanics-index`，定命名查 `glossary`
- **文档即 API**：文档之间通过 `M-*` / `VS-*` / `TASK-*` ID 形成显式引用网络，可被校验脚本自动检查
- **Progressive Disclosure**：按任务类型分级加载，不一口气塞给 agent 所有文档。2 个入口文档 + 按需加载 = 最小上下文占用
- **Feedforward + Feedback 闭环**：禁止项前置声明 → agent 执行 → 校验脚本后置检查，形成完整反馈回路
- **事后规则 > 事前预判**：最有价值的约束都来自实际事故复盘，而不是设计阶段的推演

## 技术实现

### 校验脚本（validate-workflow.mjs）

约 480 行 Node.js，15+ 条一致性规则，区分 error（阻断 CI）和 warning（提示注意）：

**文件存在性**：所有必读文件是否存在、旧路径是否已迁移

**任务看板完整性**：任务行与详细定义是否双向匹配、是否有重复 ID、Done 任务是否已迁至 `task-history.md`、Blocked 任务是否标注阻塞原因、脚手架任务是否混入游戏看板、目标引用是否包含至少一个 `M-*` 或 `VS-*`

**推荐有效性**：推荐任务是否存在且状态为 Ready、Ready 任务是否至少有一条推荐路径

**历史一致性**：`task-history.md` 的行与定义是否双向匹配、是否与活跃看板有 ID 交叉

**引用有效性**：任务引用的 `M-*` 和 `VS-*` 是否真实存在于机制表和切片表中

**依赖就绪检查**：Ready 的 TASK-ARCH/TASK-SLICE 依赖的机制逆向状态不能是"未扒"或"暂缓"

**冷启动规则**：AGENTS.md 是否声明不默认读历史、是否包含冷启动阅读分流表

**治理日志时效**：workflow 文件最近修改日期是否在 `governance-log.md` 中有对应条目

**代码质量门禁**：`package.json` 是否包含 `test:systems` / `check:code` / `check:all` scripts、`code-quality-gates.md` 是否包含必要的质量要求、`CLAUDE.md` 是否指向系统测试和质量门禁

**源码边界**：`noUnusedParameters` 开启时是否文档化了 Phaser 的 `_time` 命名约定

**领域语言**：`glossary.md` 是否有重复推荐名或字段不完整的行、`src/` 中是否使用了禁止别名

### 任务标准化

每个任务 9 个必填字段，确保新 agent 冷启动即可执行：

```
任务类型 → 目标机制/切片 → 输入资料 → 输出产物 →
完成定义 → 验收标准 → 禁止范围 → 状态更新 → 推荐后续任务
```

- 机制引用使用 `M-*` ID，切片引用使用 `VS-*` ID，不可只用中文名
- 大任务标 `Split` 后用字母后缀拆分子任务（如 `TASK-SETTINGS-004` → `004A`/`004B`/`004C`）
- 任务只能来自三个来源：机制缺口、切片缺口、工程基础设施缺口

### 任务分级路由

不做全量预加载。根据任务类型动态决定读什么：

| 任务类型 | 必读文档数 | 说明 |
|---------|-----------|------|
| 轻量请求 | 2 | AGENTS.md + TASK_OUTLINE.md |
| 正式游戏 task | 4-5 | + task-board + mechanics-index + vertical-slices |
| 玩法逆向 | 5-6 | + extracted_flash/README + AS3 路径 |
| 代码实现 | 5-6 | + src-boundaries + 对应源码 |
| 脚手架维护 | 5+ | + workflow/README + document-map + governance-log |
| 历史追溯 | +1 | + task-history（仅此时读取） |

### 代码质量门禁

为 agent 写的代码建立可自动验证的质量标准：

- 修改 `src/` → 必须运行 `npm run test:systems` + `npm run build`
- 修改文档 → 必须运行 `npm run check:workflow`
- 混合改动 → `npm run check:all`
- 命中去重、刷怪、输入、技能槽、状态机等规则更新 → 同步更新 `tools/system-tests.ts`
- 架构边界：`src/scenes/` 只做 Phaser 对象创建和调度，复杂逻辑进 `src/systems/`

### 轻量 DDD 统一语言

不引入完整 DDD，只保留统一语言、上下文边界和命名约束：

- `glossary.md` 维护每个领域概念的推荐代码名和禁止别名
- `ubiquitous-language-process.md` 定义更新流程
- 校验脚本自动扫描 `src/` 中是否出现禁止别名并报 warning
- AS3 原名作为证据记录，现代代码必须使用推荐名

## 和业界 Harness Engineering 趋势的对齐

| 业界实践 | 本项目的对应实现 |
|---------|----------------|
| Agent = Model + Harness（Mitchell Hashimoto） | 文档体系 + 校验脚本 + 质量门禁 + 统一语言 = 项目 Harness |

## 演进过程

这套脚手架不是一次性设计的，而是通过治理日志持续演进的。关键转折点：

1. **任务和脚手架分家**：最初所有任务（包括文档治理）都挤在一个看板里。将 `TASK-DOCS-*` 移出游戏看板，脚手架维护单独记录到 `governance-log.md`
2. **看板和历史拆分**：Done 任务不删、不隐藏，而是迁移到 `task-history.md`——保留可追溯性，但不占用新对话上下文
3. **校验从无到有、从粗到细**：最初的校验只检查文件在不在。逐步加入了双向匹配、M/VS 引用、Ready 依赖就绪、治理日志时效、代码质量门禁、领域语言等多维度
4. **从"读 6 个文档"到任务分级路由**：最初设计让所有 agent 读 6 个核心文档。发现轻量请求（改个常量、问个配置）不需要读机制表和切片表——把上下文留给实际工作更划算。改造为三档分级 + 阅读分流表
5. **代码质量门禁**：视觉测试不够可靠，为 agent 建立可自动验证的质量标准（`test:systems` + `build` + `check:workflow`）
6. **统一语言**：agent 给同一个概念起不同的变量名，导致代码库不一致。引入轻量 DDD，校验脚本自动扫描禁止别名
7. **对话收束规则**：从反复踩坑中提炼——agent 做了少量工作就建议"新开个对话"，实际应该 compact 继续

核心教训：事前预判的约束价值远低于事后从事故中凝练的约束。靠推演写的"不要一次性重构"，agent 还是会犯。但"方向键归 P2"这种从一次具体事故中提取的规则，agent 几乎不会再违反。现在我会刻意在每次 agent 犯错后立即更新约束，而不是试图提前想到所有坑。

## 面试问答

### Q: 为什么不直接用 Jira/Linear 管理任务？

文档和代码在同一仓库，agent 可以直接读取。Jira 需要网络请求 + 权限 + 格式解析，增加了 agent 的失败点。文件系统是最可靠的 agent 数据源——"如果不在仓库里，对 agent 来说就不存在。"

而且文件系统天然支持校验脚本——校验脚本可以直接 parse Markdown 表格、提取 ID、做交叉验证。换成 Jira API 就复杂多了。

### Q: 文档维护成本会不会很高？

校验脚本兜底。文档不一致会直接报 error，阻断 CI。每次任务完成后必须更新对应文档，这是完成定义的一部分。治理日志单独维护，不和游戏任务混在一起。

另外，任务分级路由降低了轻量请求的文档读取成本——大部分日常交互只读 2 个入口文档，不需要翻阅全部文档。

### Q: 这套东西能迁移到其他项目吗？

核心框架（任务分级路由 + 标准化任务模板 + 校验脚本 + 质量门禁 + 治理日志）是领域无关的。需要替换的部分是：机制索引 → 项目独有领域的术语表或 API 契约表，纵向切片 → 端到端功能模块表。

迁移到后端项目、前端 App、CLI 工具都可以复用同一套 Harness 骨架。

### Q: AGENTS.md 和 CLAUDE.md 有什么区别？为什么要有两个入口？

`CLAUDE.md` 是 Claude Code（一个具体 AI 工具）的快速入口，摘要列出启动校验、核心约束和代码质量门禁，约 30 行。`AGENTS.md` 是所有 AI agent 的通用协作入口，包含任务分级、阅读分流表、对话收束规则等，约 60 行。

之所以分开，是因为 Claude Code 在每次对话开始时会自动加载 `CLAUDE.md`，其他工具不一定。所以把最关键的启动校验和约束放在 `CLAUDE.md`，把更详细的协议放在 `AGENTS.md`，两个文件之间有指针互引。

### Q: "面包屑"是什么意思？

面包屑（Breadcrumbs）是来自 Context Engineering 的概念。在 agent 脚手架中，它指**入口文件不包含完整信息，只放指向深层文档的路径指针**，让 agent 知道"遇到什么问题应该去读哪个文件"。

比如 `CLAUDE.md` 的冷启动阅读分流表不写"五个角色的普攻连段是什么"，而是写：

```
如果任务涉及玩法或实现进度，去读：
- docs/tasks/task-board.md
- docs/reverse-engineering/mechanics-index.md
- docs/tasks/vertical-slices.md
```

这和业界总结的 **"Map, Not Manual"** 原则一致——入口文件是一张地图，不是一本百科全书。一本大而全的入口文件有三个问题：
1. 挤占上下文窗口——agent 的上下文是有限资源
2. 维护成本高——谁改了什么很难追踪
3. 无法按需加载——做代码实现的 agent 不需要读脚手架维护规则

### Q: 任务分级路由和传统"启动必读清单"有什么区别？

传统做法是给出一份固定清单：无论什么场景，agent 启动就按顺序读 6-8 个文档。这在所有任务都是正式游戏 task 时没问题，但现实中大量交互是"这个配置什么意思""帮我改个常量"这种轻量请求。

分级路由的核心洞察：**上下文窗口是有限资源，不是所有任务需要同等的背景知识**。所以设计了冷启动阅读分流表——按 8 种任务类型给出最小必读集。轻量请求 2 个文档，正式 task 4-5 个，逆向/实现 5-6 个。这是 Progressive Disclosure 在文档层面的落地。

### Q: 最大的教训是什么？

其中一个是，**"建议新开对话"这个看似无害的建议，实际上浪费了大量上下文**。agent 做完一个小修改就建议"建议新开对话继续"，用户照做了，新 agent 又要重新读文档、重新理解当前状态。这个问题从事后复盘发现后，我在 `AGENTS.md` 中加了一条具体规则：同一个 task 未完成时默认继续当前对话，上下文过长时优先 compact，不要因为只做了少量工作就建议切换。

另一个教训是**约束必须具体到行为级别**。`AGENTS.md` 里写"不要一次性重构"，agent 理解不了什么叫"一次性"。但写"每个禁止项下列出明确的、可被验证的具体行为"，比如"不把方向键分配给玩家 1"，agent 就能严格遵守。

### Q: 轻量 DDD 在 agent 协作中解决了什么实际问题？

在没有统一语言之前，不同的 agent（甚至同一个 agent 在不同对话中）会给同一个概念起不同的变量名——比如玩家输入状态有的叫 `playerInput`，有的叫 `inputState`，有的叫 `controlState`。这导致代码库可读性下降，后续 agent 更难理解代码。

轻量 DDD 的做法：建一张表（`glossary.md`），每个领域概念一行，写明推荐代码名和禁止别名。校验脚本自动扫描 `src/` 中是否出现禁止别名并报 warning。不追求完整 DDD 的聚合根/值对象/领域事件等重型概念，只取"统一语言"这一个最有实际价值的实践。

## 量化参考

- 校验脚本从约 200 行演进到约 480 行，覆盖 15+ 个校验维度，区分 error/warning 两级
- 文档体系包含 20+ 个职责正交的 Markdown 文件，通过 `M-*` / `VS-*` / `TASK-*` ID 形成可校验引用网络
- 治理日志记录了 10+ 次脚手架迭代，每次含日期、变更内容、影响范围、验证结果
- 从最初的"所有任务混在一起"演进为四层分离：游戏任务层 / 逆向笔记层 / 领域语言层 / 脚手架治理层
- 代码质量门禁覆盖三层验证链：`check:workflow` → `test:systems` → `build`
- 冷启动一个全新 agent 会话做轻量请求：读 2 个入口文档（< 2 分钟）即可独立回答问题
- 冷启动做正式游戏 task：读 4-5 个文档（约 5 分钟），即可独立选择并执行一个 Ready 任务
- 对话不需要人为"交接"——agent 按阅读分流表自行加载所需上下文，文档的 ID 引用网络保证信息一致性
