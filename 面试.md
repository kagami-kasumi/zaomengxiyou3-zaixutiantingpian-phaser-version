# 面试准备：Harness Engineering 与长期 AI Agent 协作

## 这份材料怎么用

这份材料专注于可迁移到不同项目的 Harness Engineering、Agent 协作、工程治理和工具集成。游戏逆向与具体系统实现见 `游戏-面试.md`。

- 30 秒：讲清长期 Agent 协作的问题、方案和结果。
- 2 分钟：展开外部记忆、执行边界、质量反馈和演进方法。
- 深挖：进入上下文工程、Feature Line/Goal/Task、问题治理、自动校验或 MCP/LSP。

表达时不要把 Harness 说成“一组更长的 Prompt”。重点是如何把目标、状态、边界、证据和反馈做成可执行的工程系统。

---

## 30 秒概述

我在一个长期 TypeScript 遗留系统现代化项目中，设计并持续迭代了一套面向 AI Agent 的协作 Harness。项目跨逆向研究、架构设计、代码实现、视觉验收和文档治理，不同任务可能由不同对话接手，主要风险是上下文丢失、范围失控、状态漂移和误报完成。

我的方案是把项目知识做成按需加载的外部记忆，把交付拆成 Feature Line、Goal 和 Task 三层边界，再用结构检查、系统测试、构建和工作流校验形成反馈闭环。我还通过 MCP 接入 TypeScript LSP，让 Agent 能用语义引用和 diagnostics 辅助判断影响范围。

一句话总结：**我做的不是让 Agent 记住更多内容，而是让它能够恢复状态、在明确边界内行动，并用机器反馈证明结果。**

---

## 背景与我要解决的问题

单轮代码生成并不难，真正困难的是长期协作：

- 每个新对话都可能从零开始，没有可靠的长期记忆。
- 项目知识分散在代码、任务、历史决策、外部资料和用户反馈中。
- Agent 容易扩大范围、跳过前置研究或顺手重构无关系统。
- 一个小任务完成后，可能被误报成完整功能已经交付。
- 文档、代码、测试和任务状态可能彼此矛盾。
- 上下文不断增长后，Agent 可能因为压缩或切换而丢失关键约束。
- 纯文本搜索难以可靠判断大型 TypeScript 项目中的符号关系。

因此 Harness 需要同时回答五个问题：

1. 新 Agent 应读取什么，而不是把什么都读一遍？
2. 当前允许做什么，明确不能做什么？
3. 做到什么程度才算完成？
4. 如何自动发现越界、不一致和回归？
5. 下一轮如何恢复事实、决策和剩余工作？

## 核心模型

```text
入口与上下文路由
        │
        ▼
分层外部记忆
目标 ── 当前状态 ── 历史 ── 证据 ── 统一语言
        │
        ▼
执行边界
Feature Line ── Goal ── Task
        │
        ▼
实现与反馈
代码/文档变更 ── 结构检查 ── 测试 ── 构建 ── 状态回写
        │
        └──────── 发现系统性问题时进入问题治理
```

这套设计有六个原则：

1. **入口是地图，不是百科全书**：入口只保留高频硬规则和阅读路由。
2. **上下文按任务加载**：轻量请求、正式实现、评审、逆向和治理读取不同资料。
3. **交付承诺与工作单元分开**：Task 完成不自动等于完整功能完成。
4. **事实、推断和设计选择分开**：Agent 不能把证据缺口补写成确定结论。
5. **规则同时需要前置约束和后置反馈**：自然语言负责引导，脚本和测试负责纠偏。
6. **约束边界，不微管理实现**：明确目标、禁止范围和验收方式，把实现空间留给 Agent。

## 五个关键设计

### 1. 分层外部记忆：解决跨对话失忆

#### 问题

如果所有背景都塞进一个入口文件，冷启动成本会持续增长，内容也会快速过期；如果完全依赖聊天历史，新对话又无法恢复状态。

#### 方案

我把信息按职责拆开：

| 信息层 | 回答的问题 |
| --- | --- |
| 战略导航 | 项目目标、阶段路线和基本原则是什么 |
| 交付台账 | 当前完整承诺、状态和关闭条件是什么 |
| Goal 台账 | 本轮只交付什么，范围和上下文预算是多少 |
| Task 看板 | 当前最小工作单元、依赖和验收是什么 |
| 历史记录 | 过去完成了什么，为什么这样决策 |
| 证据索引 | 某个结论是否已经研究清楚、是否完成实现 |
| 架构与统一语言 | 代码边界和推荐命名是什么 |
| 工作流协议 | 不同类型任务如何执行、验证和收尾 |

入口文件不复制这些内容，只负责路由：

- 轻量问题只读入口和直接相关文件。
- 正式实现才读取交付、Goal、Task、证据和架构资料。
- 评审与问题治理加载各自协议。
- 历史文件只在追溯决策或修改已完成工作时读取。

#### 价值

这相当于建立了一个可查询的项目记忆，而不是要求 Agent 一次性记住整个仓库。上下文体积可控，文档职责也更容易验证。

### 2. Feature Line / Goal / Task：解决范围失控和误报完成

三个层级回答不同问题：

```text
Feature Line：对用户或业务的完整交付承诺
  └─ Goal：一次对话或一次自动推进的有界交付包
      └─ Task：最小可独立验收的工作单元
```

- **Feature Line** 持有完整范围、关闭矩阵和最终证据。
- **Goal** 限制单次执行范围、绑定 Task，并约束上下文压缩预算。
- **Task** 包含输入、产物、完成定义、验收、禁止范围和状态更新。

项目保持单条 Feature Line `WIP=1`。Task 或 Goal 完成后仍在同一条线上连续推进，只有完整关闭合同满足后才切换系统。

这个模型避免了两种常见问题：

- 完成一个局部 Task 就宣称整个功能完成。
- 一次对话不断追加工作，最终超出可验证和可交接的范围。

### 3. 证据合同：解决 Agent 在资料不足时边猜边写

对于遗留系统、外部规范、复杂业务规则或视觉复现，找到相关文件不等于结论已经成立。

我要求关键结论至少记录：

- 一手来源和实际调用链；
- 输入、状态、触发条件、边界和可观察结果；
- 已确认事实、推断、未知和现代设计选择；
- 现代实现由哪个 owner 消费；
- 自动验证和运行验证分别证明什么；
- 哪些反证会让结论重新进入调查。

这套方法不只适用于游戏逆向，也可以迁移到：

- 遗留业务系统迁移；
- 第三方 API 行为兼容；
- 数据口径和报表重构；
- 复杂权限与审批流程；
- 设计系统和交互还原。

关键思想是：**先把来源材料转成与架构无关的行为合同，再决定现代实现。**

### 4. 前置约束 + 后置反馈：解决“规范写了但没人遵守”

前置约束负责告诉 Agent 应该做什么：

- 必读资料与按需阅读路由；
- 当前 Feature Line、Goal 和 Task；
- 可写范围和禁止范围；
- 完成定义与验证命令；
- 何时必须停止扩展并交接。

后置反馈负责发现偏差：

- 工作流校验检查文档、引用和状态是否一致；
- 结构检查阻止继续向热点文件堆逻辑；
- 专项测试验证确定性业务规则；
- 类型检查和构建验证完整工程；
- 运行时验收验证无法由静态测试证明的行为；
- 搜索或审计确认旧定义、重复数据源和废弃路径已经消失。

核心原则是：**能够机械判断的规则，应尽量变成脚本、测试、Schema 或构建失败。**

### 5. 问题治理：解决系统性缺口被当成零散建议

普通评审很容易留下“建议抽象”“职责有点重”“测试不够”之类无法关闭的意见。

我把会持续伤害系统演进能力的问题升级为治理对象，例如：

- 跨文件重复；
- 模块耦合和边界漂移；
- 单一数据源缺失；
- 生命周期规则不统一；
- 测试盲区；
- 文档、任务和代码状态分裂。

每个治理对象必须包含：

| 部分 | 必须回答的问题 |
| --- | --- |
| 问题定义 | 为什么它是系统性问题 |
| 证据 | 哪些文件、调用链、重复或事故能证明 |
| 解决方案 | 治理到什么边界，明确不做什么 |
| 测试方案 | 如何证明方案有效 |
| 测试结果 | 实际执行结果和剩余风险 |
| 关闭标准 | 什么情况下才允许关闭 |
| 效果反馈 | 后续任务是否复发，方案是否需要退回 |

评审负责发现和分级，问题治理负责让系统性问题可确认、可执行、可验证、可关闭。

## 自动校验与质量门禁

当前验证链按风险分层：

| 改动类型 | 主要验证 |
| --- | --- |
| 工作流、任务或领域文档 | 工作流与引用一致性校验 |
| 在现有源码中新增逻辑 | 修改前先运行结构检查 |
| 业务系统实现 | 专项测试 + 全量系统测试 |
| 类型和打包边界 | TypeScript 检查 + 生产构建 |
| 跨层混合改动 | 完整门禁 |
| 视觉、交互或时序 | 自动检查 + 运行时观察 |

工作流校验脚本已从早期的基础文件检查演进到约 1200 行，覆盖：

- 必读文件和必要命令是否存在；
- Feature Line、Goal、Task 是否保持唯一激活状态；
- Task 详情与看板是否双向一致；
- 已完成工作是否迁移到历史；
- 稳定 ID 和跨文档引用是否真实存在；
- 前置证据是否满足实现条件；
- 工作流变更是否有治理记录；
- 统一语言是否出现重复推荐名或禁止别名；
- 问题治理是否完成触发条件和效果反馈扫描。

最重要的不是校验规则数量，而是把“文档是否可信、状态是否可执行”从人工印象变成可重复结果。

## MCP + LSP：从文本搜索升级到语义导航

文本搜索适合快速发现候选，但不知道某个命中是符号引用、同名变量、注释还是字符串；面对路径别名、barrel export 和 re-export，仅靠关键词容易漏改或误改。

因此我给 Codex 接入了基于 MCP 的 TypeScript LSP：

```text
Codex（MCP Host）
  → codex-lsp（MCP Server）
  → typescript-language-server（LSP Server）
  → TypeScript 项目与 tsconfig
```

- MCP 负责把外部能力暴露为可发现、带 Schema 的工具。
- LSP 负责 diagnostics、definition、references、symbols 和 rename 等语言语义。
- 构建与测试继续负责全项目类型和运行行为，LSP 不替代它们。

实际验证分为四层：

1. MCP 初始化成功。
2. 工具能够被 Host 发现。
3. 下游语言服务器能够启动。
4. 真实项目上的 diagnostics、定义和引用查询有效。

工程集成中还处理了 GitHub ZIP 不包含 submodule，以及 Windows 下无 shell `spawn` 不能稳定执行 npm `.cmd` 包装器的问题，最终改为直接启动 `node.exe + typescript-language-server/lib/cli.mjs`。

这部分不是从零实现 MCP 或 LSP 协议。我的工作是识别纯文本理解的风险、选择并集成现有组件、解决 Windows 启动链问题，并证明端到端语义调用有效。

分层使用方式是：

```text
文本搜索：发现候选位置
LSP：确认符号关系与类型诊断
编译和测试：验证完整工程与行为
```

## 这套 Harness 如何演进

它不是一次性设计完成的，而是从真实失败中逐步增加约束：

1. **入口从百科全书变成路由**：固定必读清单导致上下文浪费，后来按任务类型加载。
2. **活跃状态与历史分离**：完成记录移入历史，活跃看板只保留当前执行信息。
3. **Task 与完整交付分离**：Task 完成曾被误解为系统完成，因此增加 Feature Line 关闭合同。
4. **增加有界 Goal**：避免一次自动推进跨越多个工作包，并显式管理上下文压缩预算。
5. **从文档规则升级为自动门禁**：引用、状态、前置依赖和命名进入校验脚本。
6. **从评审建议升级为问题治理**：系统性问题必须有证据、方案、测试和反馈闭环。
7. **逆向从“找到文件”升级为证据链**：局部资料必须追到共享调用链、行为合同和验证。
8. **代码理解从 grep 扩展到 LSP**：文本候选之后用语义工具确认影响范围。

最大的体会是：**高价值规则通常来自具体事故，应该沉淀成路由、数据边界、测试或校验，而不是继续把 Prompt 写长。**

## 对其他场景的适配

这套方法不是游戏专用。可复用的是“外部记忆、执行边界、证据合同和反馈闭环”，需要替换的是领域索引与验收方式。

| 场景 | 保留的 Harness | 需要替换的领域部分 |
| --- | --- | --- |
| 遗留系统迁移 | 分层记忆、证据合同、Goal/Task、回归门禁 | 旧接口、数据库行为、兼容矩阵 |
| 企业业务平台 | Feature Line、统一语言、问题治理、权限边界 | 业务流程、角色、审计和合规规则 |
| 数据工程 | 数据源证据、单一口径、依赖门禁、回归测试 | Schema、血缘、质量指标和重跑策略 |
| SDK/API 集成 | 调用链、未知分级、端到端验证 | 版本矩阵、错误码、限流和兼容性 |
| 设计系统/UI 重构 | 资产索引、状态矩阵、视觉验收 | 组件、Token、响应式和无障碍规则 |
| 内容或知识库项目 | 上下文路由、来源分级、状态回写 | 引用、版权、事实更新和发布流程 |

迁移时可以保留四个最小部件：

1. 一个只做路由的入口。
2. 一个完整交付台账和一个当前工作台账。
3. 一套证据与完成定义。
4. 一条能够自动结束的验证链。

小型短期项目不需要复制完整体系。可以只保留入口、一个 Goal 文件和一条测试命令；随着真实失败出现再增加治理机制。

## 结果与可量化事实

- 建立了 Feature Line、Goal、Task、证据索引和历史记录分层。
- 全项目保持单条交付线 `WIP=1`，Task 完成不再自动代表系统完成。
- 工作流校验脚本约 1200 行，并与资源/数据校验、结构检查、系统测试和构建组成门禁。
- 当前案例仓库包含百余个职责拆分的系统模块和数十个专项测试入口，可用于检验 Harness 是否能支撑长期演进。
- 通过 MCP 接入 TypeScript LSP，增加 diagnostics、定义、引用、符号和安全重命名能力。
- 系统性问题从零散评审意见升级为带证据、验证、关闭标准和效果反馈的治理对象。
- 新 Agent 不依赖口头交接，可以按入口路由恢复目标、依赖、边界、状态和下一步。

真正重要的结果是三点：

- **可恢复**：新对话能从落盘状态继续。
- **可约束**：Agent 可以自主行动，但范围和权限明确。
- **可验证**：完成结论有机器反馈和可复查证据。

## 不应过度声称的部分

- Harness 降低错误概率，但不能保证 Agent 永远正确。
- 文档驱动不代表文档天然可信，仍需代码事实和自动一致性校验。
- 自动测试不能替代产品判断、真实用户反馈和无法自动化的运行验收。
- MCP/LSP 是现有组件的工程集成，不应包装成自研协议实现。
- `WIP=1` 适合当前强依赖、长期接力的项目；多条真正独立的交付线可以采用受控并行。
- 专业化 Agent 或 Subagent 是可选实现手段，不是 Harness 成立的前提。
- 小项目应保留核心思想并删减流程，而不是整套照搬。

## 2 分钟表达示例

我做过一套比较完整的 Harness Engineering 实践。背景是一个长期 TypeScript 遗留系统现代化项目，工作跨研究、架构、实现、评审和文档治理，而且不同对话会由不同 Agent 接手。真正的问题不是单次生成代码，而是新 Agent 会失忆、任务容易越界、局部完成容易被误报成整体完成，文档和代码状态也会逐渐漂移。

我的方案分四层。第一层是外部记忆，把目标、当前交付、历史、证据和统一语言拆成职责独立的文档；入口只做路由，按任务类型加载最小上下文。第二层是执行边界，用 Feature Line 表示完整业务承诺，用 Goal 限制一次对话的交付和上下文预算，用 Task 表示最小可验收单元。第三层是证据合同，把确认事实、推断、未知和现代设计选择分开，避免资料不足时边猜边写。第四层是反馈，把引用一致性、结构边界、系统测试、类型检查和构建做成自动门禁。

我还建立了问题治理闭环，让跨文件重复、边界漂移和数据源分裂不再停留在 Code Review 建议，而是必须有证据、方案、测试、关闭标准和后续效果反馈。在代码理解方面，我通过 MCP 接入 TypeScript LSP，让 Agent 在文本搜索之后用定义、引用和 diagnostics 确认影响范围，再由编译和测试验证行为。

这套 Harness 已经在百余个系统模块和数十个专项测试的真实仓库里持续演进。我的核心经验是：不要试图用更长的 Prompt 管住 Agent，而要把项目状态外部化、把执行范围数据化，并让可以判断的规则进入机器反馈。

## 常见追问

### Q：这和写一份很详细的 AGENTS.md 有什么区别？

详细 Prompt 仍然是自然语言建议，而且会随着项目增长变得昂贵和过时。Harness 把状态、任务、证据和历史拆成可查询的数据，并通过脚本、测试和构建反馈。入口只负责告诉 Agent 在当前场景该读取什么。

### Q：为什么不用 Jira、Linear 或 GitHub Issues？

这些工具擅长团队协作和业务追踪，但 Agent 仍需要仓库内可版本化、可全文搜索、能与代码一起审查的执行上下文。两者并不冲突：外部工具可以管理组织级计划，仓库内 Harness 负责让 Agent 获得精确、可执行的局部状态。

### Q：为什么同时需要 Feature Line、Goal 和 Task？

Feature Line 是完整业务承诺，Goal 是一次执行的范围和上下文预算，Task 是最小可验收单元。缺少这层区分时，Agent 容易完成局部任务就误报整体完成，或在一次对话中不断扩展范围。

### Q：为什么要限制单条 Feature Line WIP=1？

当前案例中的任务共享大量状态和验证链。WIP=1 可以减少跨线切换、半成品和文档竞争。它不是绝对原则；如果交付线数据、代码和验收真正独立，可以并行，但应明确所有权和合并边界。

### Q：文档维护成本不会很高吗？

会，所以文档必须职责正交、按需加载，并把一致性转成脚本。活跃状态与历史分开，避免入口不断膨胀。判断标准不是“文档越多越好”，而是它是否减少了重复调查、错误实现和交接损耗。

### Q：如何防止文档和代码事实不一致？

第一，把可检查的关系变成稳定 ID、Schema 和校验脚本；第二，完成任务时要求状态回写；第三，关键结论仍以代码、测试和运行结果为准；第四，历史只负责追溯，不反向覆盖当前事实。

### Q：问题治理和普通重构有什么区别？

普通重构可以是局部改善。问题治理针对会持续伤害演进能力的结构性缺口，必须先证明问题，再定义边界、方案、测试和关闭标准，并在后续任务中观察是否复发。

### Q：为什么需要 MCP，直接让 Agent 执行命令不行吗？

命令适合过程调用，但 Agent 还需要发现能力、理解结构化参数和返回结果，并区分只读与修改操作。MCP 提供标准工具边界；具体语义能力仍由下游服务提供。

### Q：有了 LSP，为什么还要文本搜索、typecheck 和测试？

它们回答的问题不同。文本搜索发现候选，LSP 确认定义、引用和诊断，typecheck 验证全项目类型，测试验证业务行为。任何一层都不能单独证明功能正确。

### Q：MCP/LSP 是你自己实现的吗？

不是。我集成了现有的 `codex-lsp`、`lsp-tools-mcp` 和 TypeScript language server，解决插件注册、submodule 和 Windows 进程启动问题，并完成端到端验证。工程价值在于正确选型、集成、排障和证明有效。

### Q：专业化 Agent 是不是必须的？

不是。专业化 Agent 适合研究、实现、评审或治理之间上下文和权限明显不同的场景。Harness 的核心是状态、边界和反馈；是否拆 Agent 取决于任务独立性，而不是为了多 Agent 而多 Agent。

### Q：如何把这套方法迁移到另一个项目？

先识别四件事：完整交付单位、最小工作单元、权威证据源和自动验证链。再建立一个路由入口、一个当前状态台账和一个完成合同。不要先复制所有文档，等真实失败出现后再增加历史、治理或专业化工具。

### Q：最大的教训是什么？

高价值规则通常来自具体事故。最有效的沉淀方式不是继续增加 Prompt，而是把事故转成更清晰的数据所有权、更小的执行边界、可查询的证据，或者一条会失败的自动检查。
