# 面试准备：Harness Engineering 实践

## 这份材料怎么用

这不是一篇需要逐字背诵的项目介绍，而是一组可以按面试时长展开的表达素材：

- 30 秒：讲清项目、问题、方案和结果。
- 2 分钟：补充 3 个关键设计决策和一个演进案例。
- 深挖：根据面试官的兴趣，进入校验脚本、任务路由、问题治理、Subagent 或逆向工程衔接。

表达时优先讲“为什么这样设计”和“它解决了什么失败模式”，不要只罗列文档名。

---

## 30 秒概述

我在一个 Flash 游戏现代化重写项目中，设计并持续迭代了一套面向 AI Agent 的协作脚手架。项目有约 20 万行反编译 AS3、70 多个 SWF，任务跨逆向、实现、评审和文档治理，而且不同对话可能由不同 Agent 接手。

我主要解决了三个问题：新 Agent 如何低成本恢复上下文，如何让 Agent 在任务边界内自主工作，以及如何用自动校验保证文档、任务和代码状态一致。最终形成了“分层文档记忆 + 按任务加载上下文 + 前置约束 + 后置校验 + 专业化 Agent”的闭环。

一句话总结：**我做的不是给 Agent 写一份更长的 Prompt，而是把协作规则、项目状态和质量反馈做成一套可执行的工程系统。**

---

## 项目背景与我的职责

- **项目目标**：用 Phaser 3 + TypeScript 重写 10 多年前的 Flash 游戏《再续天庭》，尽量还原外观、玩法、数值、手感和流程。
- **资料规模**：约 20 万行 AS3 反编译源码、70 多个 SWF，包含本地双人、5 个角色、战斗、装备、背包、法宝等系统。
- **协作特点**：工作跨多个 AI 对话进行，每轮对话都可能从零开始，没有天然的长期记忆。
- **我的职责**：设计并维护 Agent 的冷启动、任务编排、执行边界、质量门禁、评审与治理流程，让新的 Agent 能自行定位任务、执行、验证并留下可交接状态。

难点并不只是“让 AI 写代码”，而是让它在一个长期、资料复杂、需要持续逆向的项目中稳定协作。

## 核心设计

```text
入口与路由
AGENTS.md / CLAUDE.md
        │
        ├── 轻量请求：只加载入口和直接相关文件
        ├── 正式任务：加载任务、机制、切片与执行协议
        └── 专项工作：再加载逆向、架构、评审或治理文档
        │
        ▼
项目状态与知识
战略路线 ── 任务看板/历史 ── 机制索引 ── 纵向切片 ── 统一语言
        │                       │
        └──── M-* / VS-* / TASK-* 显式引用 ────┘
        │
        ▼
执行与反馈
标准任务定义 → Agent 执行 → 结构检查/系统测试/构建 → 状态回写
```

这套设计有五个原则：

1. **入口是地图，不是百科全书**：入口只保留硬规则和阅读路由，细节按需加载。
2. **每类文档只回答一个问题**：路线、任务、历史、机制、切片和命名分开维护。
3. **文档之间使用稳定 ID 连接**：`M-*`、`VS-*`、`TASK-*` 让关系可追踪、可校验。
4. **规则既要前置，也要有反馈**：Prompt 负责引导，脚本、测试和构建负责发现偏差。
5. **约束边界，不微管理实现**：明确禁止范围、完成标准和验证方式，把具体实现空间留给 Agent。

## 四个关键问题与解决方案

### 1. Agent 跨对话失忆：如何低成本冷启动

#### 问题

新的 Agent 不知道上一轮做了什么、为什么这样做、当前该从哪里继续。如果把全部背景都塞进入口文档，又会浪费上下文，并增加维护成本。

#### 方案：分层记忆 + 渐进式加载

我把项目状态拆成职责正交的文档：

| 层级 | 主要文档 | 回答的问题 |
| --- | --- | --- |
| 战略导航 | `TASK_OUTLINE.md` | 项目目标、阶段路线是什么 |
| 任务编排 | `task-board.md` / `task-history.md` | 现在做什么、过去做过什么 |
| 逆向索引 | `mechanics-index.md` | 机制是否扒清、是否复现 |
| 纵向切片 | `vertical-slices.md` | 可玩切片依赖哪些机制 |
| 领域语言 | `glossary.md` | 同一个概念在现代代码里叫什么 |
| 工作流 | `docs/workflow/` | Agent 应如何执行、验证和收尾 |

入口文件不复制这些内容，只告诉 Agent 在什么场景读什么。轻量请求只读入口和直接相关文件；正式游戏任务才加载看板、机制表、切片表和执行协议；历史文件仅在追溯决策时读取。

未完成任务和已完成任务也被拆开：活跃看板保持小而直接，历史仍可追溯，但不会占用每次冷启动的上下文。

#### 为什么有效

这相当于为 Agent 建立了一个“按需查询的外部记忆”，而不是要求它一次性记住整个项目。新的 Agent 只要沿入口路由读取最小必需集，就能恢复当前任务的目标、依赖、边界和完成标准。

### 2. Agent 容易越界：如何兼顾自主性与可控性

#### 问题

Agent 常见的失败模式包括：顺手重构无关系统、一次承担过多工作、在资料不足时边猜边写、照搬 AS3 的全局状态或时间轴结构，以及完成少量工作就建议切换对话。

#### 方案：前置约束 + 后置反馈

前置约束负责告诉 Agent “应该做什么、不能做什么”：

- `AGENTS.md` 只保留高频硬规则，例如不修改原始提取结果、不一次重构多个系统。
- 每个正式任务包含目标、输入、产物、完成定义、验收标准、禁止范围和状态更新要求。
- 源码边界明确：Scene 负责 Phaser 对象创建和调度，复杂战斗、关卡、技能、背包、存档和 AI 规则进入 System。
- 对话收束有明确规则：同一任务未完成时继续当前对话；上下文过长优先 compact；只在任务完成或明显切换子系统时建议新开对话。

后置反馈负责发现 Agent 没有遵守规则：

- `npm run check:workflow` 检查文档和任务状态的一致性。
- `npm run check:structure` 检查源码结构和文件职责。
- `npm run test:systems` 验证关键玩法系统。
- `npm run build` 验证类型和构建。

关键点是：**不能只靠“请遵守规范”这种自然语言要求。能够自动判断的规则，应尽量变成脚本、测试或构建失败。**

### 3. 逆向资料与现代实现脱节：如何避免边猜边写

#### 问题

项目需要从 AS3 中还原玩法事实，但现代实现又不能照搬 Flash 架构。如果逆向与实现混在一起，Agent 很容易把“原版怎么写”误当成“现代版也该怎么写”。

#### 方案：双状态机制索引 + 纵向切片

每个机制同时维护两个独立状态：

- **逆向状态**：未扒、部分已扒、已扒、暂缓。
- **复现状态**：未复现、部分复现、已复现。

一个机制可以已经研究清楚但尚未实现，也可以只实现了当前切片需要的部分。两类状态不混为一个“进度百分比”。

实现计划按可验证的纵向切片组织，每个切片用 `M-*` ID 声明依赖机制。校验脚本会检查：如果一个 Ready 任务依赖的机制仍是“未扒”或“暂缓”，则报错；只有部分研究完成时给出警告。

任务职责也被拆开：

```text
TASK-SETTINGS：只确认原版行为和证据
        ↓
TASK-ARCH：建立现代实现所需的基础设施
        ↓
TASK-SLICE：完成一个可试玩、可验证的闭环
```

这保证了“忠实记录原版行为”和“选择现代架构实现”是两个清晰阶段。

### 4. 系统性问题被当成零散 Bug：如何让问题真正关闭

#### 问题

有些风险不会立刻导致构建失败，却会持续增加维护成本，例如：同一伤害公式散落在十多个文件、相似工具函数签名不一致、一个运行时模型承载所有角色的无关字段。普通评审很容易只留下“建议优化”，之后无人跟进。

#### 方案：问题治理生命周期

我把系统性工程问题定义为会持续伤害演进能力的结构性缺口，例如跨文件重复、模块耦合、边界漂移、测试盲区、数据源分裂和生命周期规则不统一。局部 Bug、typo 或单个资源 Key 错误不升级为治理问题。

每个治理对象固定包含六部分：

| 部分 | 要回答的问题 |
| --- | --- |
| 问题定义 | 它为什么是系统性问题 |
| 证据 | 哪些文件、调用链、重复次数或测试缺口能证明它 |
| 解决方案 | 治理到什么边界，明确不做什么 |
| 测试方案 | 用什么测试、构建、搜索或人工复核证明有效 |
| 测试结果 | 实际执行结果和剩余风险是什么 |
| 关闭标准 | 什么情况下算真正关闭 |

例如，多处重复的 MP 表不是简单的“数组重复”，而是数值校准没有单一数据源。完成治理不仅要抽取共享模块、跑系统测试和构建，还要通过搜索确认旧定义已经消失。

工程评审负责发现和分级问题；问题治理负责让系统性问题从意见变成可确认、可执行、可验证、可关闭的对象。

## 关键实现

### 自动校验：把文档约定变成可执行规则

`validate-workflow.mjs` 目前有 500 余行，覆盖 15 个以上的校验维度，并区分两种级别：

- **error**：会破坏执行正确性或状态一致性，命令以非零状态退出。
- **warning**：当前可以继续，但提示潜在维护风险。

主要检查包括：

- 必读文件和必要 npm scripts 是否存在。
- 看板行与任务详细定义是否双向匹配，任务 ID 是否重复。
- Done 任务是否已迁移到历史，活跃看板与历史是否交叉。
- 推荐任务是否存在、是否为 Ready。
- `M-*` 和 `VS-*` 引用是否真实存在。
- Ready 任务依赖的机制是否已经完成足够逆向。
- 工作流文件变更是否在治理日志中留痕。
- 领域词汇表是否有重复推荐名、源码是否出现禁止别名。

这里最重要的工程选择不是规则数量，而是**将“文档是否可信”从人工印象变成自动检查结果**。

### 任务标准化：让新 Agent 拿到任务即可执行

正式任务使用统一字段：

```text
任务类型 → 目标机制/切片 → 输入资料 → 输出产物 →
完成定义 → 验收标准 → 禁止范围 → 状态更新 → 推荐后续任务
```

任务只能来自三类缺口：机制认知缺口、纵向切片缺口、工程基础设施缺口。过大的任务先标记为 Split，再拆成可独立验收的子任务。

标准化的价值不在格式整齐，而在消除隐含信息：新 Agent 不需要猜测“做到什么程度算完”或“能不能顺手改别处”。

### 代码质量门禁：按改动类型选择验证链

不同改动触发不同验证：

| 改动 | 主要验证 |
| --- | --- |
| 工作流、任务或领域文档 | `npm run check:workflow` |
| 现有源码中新增逻辑 | 先运行 `npm run check:structure` |
| 游戏系统实现 | `npm run test:systems` + `npm run build` |
| 混合改动 | `npm run check:all` |

结构检查不仅在收尾运行，也前移到修改现有源码之前：如果目标文件已经过大或职责过多，就先拆分，避免继续向热点文件堆逻辑。

### 轻量统一语言：只引入真正有收益的 DDD 部分

不同 Agent 可能为同一概念创造不同名字，导致代码逐轮漂移。项目用 `glossary.md` 维护推荐代码名、原版证据名和禁止别名，并用脚本扫描明显冲突。

这里没有引入完整的聚合根、领域事件等 DDD 体系，只采用统一语言和边界意识。原则是：**用最小机制解决实际命名漂移，不为套用方法论增加复杂度。**

### 专业化 Subagent：让职责和权限一致

项目配置了四类专用 Agent：

| Agent | 主要职责 | 写入边界 |
| --- | --- | --- |
| `reverse-engineering-researcher` | 查找 AS3 证据、确认机制事实 | 只读 |
| `modern-implementation-engineer` | 用 TypeScript/Phaser 实现已确认机制 | 当前任务需要的源码、测试和状态文档 |
| `engineering-reviewer` | 按统一协议评审代码或任务成果 | 默认只读 |
| `workflow-steward` | 维护入口、协议、校验脚本和治理规则 | 工作流脚手架文件 |

拆分依据不是职位名称，而是四种不同的认知任务：忠实取证、现代实现、独立评价、流程维护。它们需要不同的上下文和写入权限。

专业化带来三个收益：

- 逆向研究与现代架构决策不易互相污染。
- Reviewer 保持独立视角，不沿用实现者的假设。
- 每个 Agent 只加载所需规则，减少无关上下文。

并行只是附加收益，前提是任务之间不存在未满足的依赖。

## 这套系统是如何演进出来的

它不是一次性设计完成的，而是从真实失败中逐步加规则：

1. **任务与脚手架分家**：最初游戏任务和文档治理混在一个看板，后来将工作流维护迁入独立治理日志。
2. **看板与历史分离**：完成任务迁入历史，既保留追溯能力，又控制默认上下文体积。
3. **从固定必读清单改为任务分级路由**：轻量请求不再加载机制表、切片表等无关资料。
4. **校验从“文件是否存在”扩展到交叉一致性**：逐步增加任务双向匹配、引用有效性、依赖就绪和领域语言检查。
5. **质量检查前移**：不仅在完成后测试，还在向现有文件添加逻辑前检查结构风险。
6. **增加对话收束规则**：解决 Agent 完成少量工作就建议另开对话、造成重复冷启动的问题。
7. **建立问题治理协议**：让评审中的系统性发现有证据、有边界、有关闭标准。

我最大的体会是：**高价值规则通常来自具体事故，而不是提前写出的抽象“最佳实践”。**

例如，“保持任务聚焦”太抽象；“一次只完成一个 task”“不得修改原始提取目录”“方向键属于玩家 2”则可以直接判断是否违反。每次出现新的失败模式，就把它沉淀为路由、边界、检查或测试中的一种，而不是继续把 Prompt 写长。

## 结果与可量化事实

- 形成 20 多个职责分离的 Markdown 文档，通过 `M-*`、`VS-*`、`TASK-*` 建立显式引用。
- 工作流校验脚本超过 500 行，覆盖 15 个以上的校验维度，并区分 error 与 warning。
- 建立 `check:workflow`、`check:structure`、`test:systems`、`build` 等分层验证入口。
- 将游戏任务、逆向资料、领域语言、工作流治理拆成独立层，减少状态混写。
- 新 Agent 不依赖人工口头交接，可按入口路由恢复任务目标、依赖、边界和状态。
- 系统性工程问题从零散评审意见升级为带证据、验证和关闭标准的治理对象。

这些结果中，最重要的不是文档或脚本的数量，而是协作流程具备了三个性质：**可恢复、可约束、可验证。**

### 不应过度声称的部分

- 文档驱动不等于文档永远正确，因此仍需要自动一致性检查和代码事实校验。
- Harness 降低了 Agent 出错概率，但不能替代玩法实测、视觉校准和人的产品判断。
- 专业化 Agent 适合边界清晰的任务；强依赖、频繁共享状态的工作不应为了“多 Agent”强行并行。
- 当前体系针对该项目的长期协作成本优化过，迁移到小型、短期项目时应删减，而不是整套照搬。

## 2 分钟表达示例

我做过一个比较完整的 Harness Engineering 实践。项目是把一个老 Flash 游戏用 Phaser 3 和 TypeScript 重写，资料里有约 20 万行反编译 AS3 和 70 多个 SWF。真正困难的地方不是单次生成代码，而是多个 AI 对话长期协作时，每个新 Agent 都会失忆，而且容易越界、猜测实现或留下不一致状态。

我的方案分三层。第一层是外部记忆：把路线、任务、历史、机制、纵向切片和领域语言拆成职责独立的文档，用稳定 ID 串起来。第二层是上下文路由：入口文件只做地图，轻量问题只加载少量文件，正式任务再按类型加载对应协议和资料。第三层是执行反馈：任务有明确的输入、产物、验收和禁止范围，完成后由工作流校验、结构检查、系统测试和构建发现偏差。

项目还把逆向、实现、评审和流程维护拆成专业化 Agent。逆向 Agent 只负责忠实取证，实现 Agent 用现代架构重写，Reviewer 独立检查，流程 Agent 维护 Harness。对于跨文件重复、边界漂移这类不会立即构建失败的问题，我又增加了问题治理协议，要求每个问题都有证据、方案、测试和关闭标准。

这套系统不是一次设计出来的。很多关键规则都来自真实失败，比如完成任务和历史混在一起导致冷启动变重、所有请求固定读六个文档浪费上下文、Agent 做一点工作就建议新开对话。我的做法是把每次事故沉淀成可执行的路由、边界或检查，而不只是继续增加 Prompt。

## 常见追问

### Q：为什么不用 Jira 或 Linear 管任务？

这个项目的主要执行者是仓库内工作的 Agent。Markdown 与代码同版本、可离线读取、可直接被脚本解析，失败面更小。任务状态还能与机制 ID、切片 ID 和代码变更一起校验。

这不是说 Jira 不好。如果项目需要跨团队排期、权限、通知和管理报表，外部平台仍有价值；但关键执行上下文最好在仓库内保留一份机器可读的事实源。

### Q：文档维护成本不会很高吗？

会，所以设计重点不是“多写文档”，而是控制职责和读取成本：每份文档只回答一个问题，已完成任务移入历史，轻量请求不加载完整体系。对高风险一致性则用脚本兜底。

如果某条信息无法明确归属、长期无人读取、也不能参与决策或校验，就不应该新增文档。

### Q：为什么 AGENTS.md 和 CLAUDE.md 要分开？

`AGENTS.md` 是跨工具的通用协作入口，包含任务分级、阅读路由和收束规则；`CLAUDE.md` 是 Claude Code 会自动发现的快速入口，保留启动检查、核心约束和专用 Agent 导航。两者互相引用，但不复制完整协议。

本质上是适配不同工具的发现机制，同时让核心规则有唯一来源。

### Q：所谓“面包屑”或“入口是地图”是什么意思？

入口文件不承载全部知识，只告诉 Agent “遇到哪类任务，到哪个文件取事实”。例如代码实现会指向源码边界和对应机制文档，历史追溯才指向任务历史。

这样做既节省上下文，也让知识的归属和更新位置更清楚。

### Q：任务分级路由与固定必读清单有什么区别？

固定清单假设所有任务都需要同样多的背景。分级路由把上下文看成有限预算：解释配置、修正文案和执行正式游戏任务所需的信息量不同，因此只加载完成当前任务所需的最小集合。

### Q：为什么需要四个专用 Agent，而不是一个全能 Agent？

因为四类工作有不同的事实标准和写入边界。逆向要求忠实记录原版，现代实现要求主动舍弃旧架构，评审要求独立判断，流程维护关注的是协作系统本身。

拆分能减少上下文串扰并限制误写，但不是 Agent 越多越好。边界不清或高度耦合的工作仍由一个主 Agent 连续完成更合适。

### Q：评审协议与普通 Code Review 有什么区别？

普通 Review 容易依赖个人偏好。这里先固定判断基准和优先级：先看行为与测试，再看结构边界和任务状态，最后看命名风格；问题使用统一严重度，并明确整改应该直接修、进入游戏任务，还是升级为治理问题。

它的价值是让不同 Agent 的评审结论可比较、可追踪、可转化为后续动作。

### Q：问题治理与普通重构有什么区别？

重构关注代码如何变好；问题治理关注一个系统性风险如何被证明、限定范围、验证并关闭。局部代码“不够优雅”不自动进入治理流程，只有会持续伤害扩展能力的问题才值得承担这套成本。

### Q：为什么只用了轻量 DDD？

项目的真实问题是跨 Agent 的命名漂移，而不是缺少完整的领域建模体系。因此只引入统一语言、推荐名和边界约束，避免为方法论本身增加过多抽象。

### Q：为什么没有为每个流程配置自定义 Skill 或斜杠命令？

当前自然语言任务加阅读路由已经能稳定触发流程，专用 Agent 也覆盖了权限和上下文隔离。Skill 的主要增量是调用便捷性，而不是新的工程能力。

如果未来使用者增多、对协议不熟悉，或者某些流程需要频繁重复调用，再把成熟流程封装为 Skill 更合适。

### Q：这套 Harness 能迁移到其他项目吗？

可以迁移的是骨架：入口路由、分层事实源、标准任务、自动校验、质量门禁和治理闭环。需要替换的是领域部分，例如把游戏机制索引换成 API 契约、数据管道或业务能力表，把可玩切片换成端到端功能切片。

迁移时应根据项目周期和风险做减法。短期小项目可能只需要入口、任务模板和验证命令，不需要完整治理体系。

### Q：最大的教训是什么？

第一，具体、可验证的行为边界比抽象原则有效。第二，上下文不是越多越好，路由和删除同样重要。第三，能通过脚本、测试或权限表达的规则，不应只停留在 Prompt。第四，Harness 必须从实际失败中演进，否则很容易变成形式完整但没有反馈能力的文档工程。
