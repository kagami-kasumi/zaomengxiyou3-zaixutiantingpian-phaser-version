# 《再续天庭》EXE、Enigma 与 SWF 资源逆向技术讲解

## 1. 先说结论

`再续天庭1.1.exe` 中确实保存着角色包、怪物包和关卡包的原始数据。资源被 Enigma Virtual Box 封装进 EXE 的虚拟文件系统，之前按 `FWS`、`CWS`、`ZWS` 文件头扫描得到的数字 SWF 候选并不是可靠的原始文件清单。

2026-07-15 已使用 `evbunpack` 0.2.6 完成实验和全量恢复：EVB 文件系统恢复 206 个文件，175 个 SWF 中 174 个可还原为标准 `FWS/CWS/ZWS` 并由 FFDec 解析。详细命令、哈希、异常项和视觉抽检见 [`evb-extraction-report.md`](evb-extraction-report.md)。

## 2. 工具下载地址

推荐先使用开源工具 `evbunpack`：

- GitHub 项目主页：<https://github.com/mos9527/evbunpack>
- GitHub Releases：<https://github.com/mos9527/evbunpack/releases>
- PyPI 项目页：<https://pypi.org/project/evbunpack/>

截至 2026-07-15，PyPI 最新版本为 `0.2.6`，发布于 2025-11-02。GitHub 仓库已于 2026-02-06 被作者归档为只读。工具说明宣称支持恢复 Enigma 虚拟文件系统，包括内置文件、外部包和压缩模式，并提供只列目录的 `-l` 参数。

优先从上述 GitHub Releases 或 PyPI 下载，不要使用不明软件下载站。下载后先用 Windows Defender 扫描；如果使用 PyPI 文件，可以在 PyPI 的文件详情页核对 SHA-256。

项目规则不允许 Agent 自行下载 GitHub 软件，因此下载和准备工具需要人工完成。工具准备好后，列表检查、解包命令、结果验证、SWF 还原和 FFDec 导出可以交给 Agent。

## 3. Enigma Virtual Box 是什么

这里的 Enigma 指 **Enigma Virtual Box**，不是密码学中的恩尼格玛机，也不是 Oracle VirtualBox 虚拟机。

它解决的是“程序发布时文件太多”的问题。开发者原本可能有这样的目录：

```text
game.exe
assets/
  WuKong.swf
  TangSeng.swf
  levels/
    level11.swf
  ...
```

Enigma Virtual Box 可以把主程序和这些依赖文件打包成一个 EXE。程序运行时仍然以为自己在读取：

```text
./assets/WuKong.swf
```

但 Enigma 的加载器会从 EXE 内部的虚拟文件系统中找到对应数据，再向程序提供文件内容。文件不一定作为普通文件出现在磁盘上。

可以把它理解为：

```text
普通目录                         Enigma 封装后

game.exe                         再续天庭1.1.exe
assets/WuKong.swf       ──►      ├─ 原程序
assets/level11.swf               ├─ Enigma 加载器
其他 DLL、图片、配置             └─ 虚拟文件容器
```

这种技术首先是部署和便携化工具，不必然意味着恶意加密。不过，压缩、虚拟路径和文件头混淆也会让普通解压软件、FFDec 和基于文件头的扫描无法直接得到原文件。

## 4. 本项目已经确认了什么

### 4.1 EXE 确实包含 Enigma 容器

只读分析得到以下 PE 节：

| PE 节 | RawSize | 含义 |
| --- | ---: | --- |
| `.text` | 43,520 bytes | 普通程序代码的一部分 |
| `.rsrc` | 69,632 bytes | Windows 资源 |
| `.enigma1` | 473,612,288 bytes | 主要 Enigma 封装数据 |
| `.enigma2` | 311,296 bytes | Enigma 相关数据/加载结构 |

原 EXE 大小为 474,075,136 bytes，绝大部分体积都落在 `.enigma1`。这比“EXE 只是一个 8 MB 主 SWF”的解释更符合现有证据。

### 4.2 EXE 中存在完整的资源路径表

从 EXE 原始字节中能识别出至少 170 条唯一的 `assets/...swf` 路径，其中包括：

```text
assets/WuKong.swf
assets/TangSeng.swf
assets/BaJie.swf
assets/ShaShen.swf
assets/bailongSword.swf
assets/Role1Effect.swf
assets/Monster1111.swf
assets/levels/level11.swf
```

这是非常重要的证据。AS3 中的加载路径是运行时拼接的：

```actionscript
AssetsUrl.PATH + "assets/" + resourceName + ".swf"
```

而 EXE 中还能找到完整的 `assets/WuKong.swf` 字符串，说明它不只是 AS3 代码中的零散单词，更可能来自 Enigma 的虚拟文件目录或应用文件清单。

相关代码见：

- `local-resources/regima/legacy-extraction/resources_by_swf/[172845].swf/scripts/loader/AssetsUrl.as`
- `local-resources/regima/legacy-extraction/resources_by_swf/[172845].swf/scripts/loader/AssetsLoader.as`

### 4.3 AS3 已经告诉我们资源是干什么的

AS3 能提供的是语义和行为关系，例如：

- `WuKong` 是角色资源包；
- `Role1Effect` 是角色 1 的效果包；
- `levels/level11` 是第 1 大关第 1 小关的关卡资源包；
- `Role1Bullet1` 在哪种攻击中创建、如何跟随角色、何时移除；
- `sl11`、`bg11`、`floorBg1` 分别被哪类场景对象使用。

所以“真素材缺失”并不等于“AS3 不知道图片用途”。更准确的说法是：

> 语义名称和用途已经知道，但当前工作区尚未取得名称对应的 MovieClip、时间轴、矢量形状和位图帧。

### 4.4 为什么之前得到的是数字 SWF

此前的方法在整个 EXE 中寻找看起来像 SWF 的字节序列，并以位置或候选编号命名。这种方法适用于“多个标准 SWF 直接拼在一个文件里”的情况，不适合压缩的虚拟文件容器。

容器内部的压缩数据可能偶然出现 `FWS`、`CWS` 或 `ZWS` 字节。以它们为起点截取后，FFDec 可能会：

- 把随机字节解释成极大的帧数；
- 出现大量 `Unknown` 标签；
- 导出一个图片后便读到错误结构；
- 长时间尝试导出实际上不存在的时间轴。

例如当前 `[223622265].swf` 的只读标签统计包含 183 个 `Unknown` 和 9 个 `Error`。因此它所谓的 30,652 帧不能作为“里面有完整角色动画”的证据。

### 4.5 UI 缺失问题是怎样被发现的

这次突破并不是先知道“答案是 Enigma”，而是在长期 UI 检索中逐步发现旧解释无法同时满足所有证据。

#### 第一阶段：把主 SWF 当成完整素材库

最初通过 FFDec 从 EXE 扫出一批数字命名的 SWF，并建立了 `local-resources/regima/legacy-extraction/resources_by_swf`。这批资料非常有价值：主 AS3、部分图片、shape 和 SymbolClass 都能读取，足以逆向界面入口、按钮行为和资源名称。

但在真正寻找 UI 时出现了持续矛盾：

- AS3 明确存在背包、技能、商店、装备、法宝等完整界面逻辑；
- 代码会按名称查询面板、按钮、图标和 MovieClip；
- 当前导出却只有零散 UI 图片，无法组成原版完整界面；
- 多次搜索 `symbols.csv`、图片目录和数字 SWF，仍找不到名称对应的时间轴。

当“代码完整、视觉资源长期缺失”反复发生在 UI、角色、怪物和关卡四个领域时，问题就不再像是几个漏导出的符号，而更像是提取边界选错了。

#### 第二阶段：发现“主包之外还有运行时资源包”

旧提取最重要的贡献不是直接给出全部 UI，而是保留了加载器 AS3。`AssetsLoader`、`Aloader` 和 `AssetsUrl` 说明游戏并不只使用一个主 SWF，而是在运行时继续加载：

```text
assets/<name>.swf
assets/levels/<stage><level>.swf
```

同一证据链还表明：

- 角色资源来自 `WuKong.swf`、`TangSeng.swf` 等独立包；
- 公共界面和功能资源也应位于独立 SWF，而不是全部嵌在主包；
- 加载完成后还会对文件前 176 bytes 重新排列，再交给 Flash `Loader.loadBytes()`。

这解释了为什么直接用 FFDec 扫主包始终找不到完整 UI：FFDec 看到的只是主程序和少量内嵌资源，没有先打开程序运行时依赖的外部资源仓库。

#### 第三阶段：文件体积和 PE 结构否定“素材已经提完”

`再续天庭1.1.exe` 约 452 MiB，而已识别的主 SWF 和零散导出远不足以解释这个体积。进一步检查 PE 节后发现，约 451.7 MiB 位于 `.enigma1`，另有 `.enigma2`；这说明绝大部分 EXE 不是普通主程序代码，而是 Enigma Virtual Box 数据。

同时，从 EXE 原始字节中能找到大量完整路径，例如：

```text
assets/WuKong.swf
assets/Role1Effect.swf
assets/levels/level11.swf
```

完整路径同时出现在“AS3 会加载它”和“EXE 原始数据记录了它”两侧，证据已经比数字 SWF 猜测强得多。由此形成新的可验证假设：

> UI 和其他真素材并未从 EXE 消失，而是作为虚拟文件被 Enigma 封装；基于 SWF 文件头的扫描绕过了虚拟文件系统目录，因此只得到偶然命中的数字候选。

#### 第四阶段：用目录恢复而不是继续猜文件头

`evbunpack -l` 首次列出了原始命名目录，随后完整提取恢复 206 个文件。与 UI 直接相关的候选包包括：

```text
assets/backpack1.swf
assets/Common1.swf
assets/EIcon1.swf
assets/EIcon2.swf
assets/fonts.swf
assets/shop.swf
assets/MagicWeapon.swf
assets/MagicWeapon2.swf
assets/StageCommon.swf
```

这一步才真正回答了“UI 在哪里”：它们不是隐藏在某个数字图片目录中等待人工命名，而是保留在原始命名的功能 SWF 包里。

#### 第五阶段：解决“文件拿到了，FFDec 仍打不开”

解包出的多数 `.swf` 开头仍不是 `FWS/CWS/ZWS`。如果没有前期 AS3 逆向，这里很容易再次误判为文件损坏。按照加载器的 176-byte 重排逻辑恢复后，175 个 SWF 中有 174 个成为标准 `FWS/CWS/ZWS`，并能被 FFDec 解析。

因此旧 `local-resources/regima/legacy-extraction` 上投入的时间并非无效：它没有直接交付完整 UI，却提供了资源语义、运行时包名和关键的字节还原算法。真正的问题是此前把“AS3 主包提取成功”误当成了“整个应用资源文件系统提取成功”。

### 4.6 现在寻找一个 UI 的完整路径

后续定位 UI 不再全目录盲扫，而是使用下面的证据链：

```text
现代界面需求
  ↓ 查 AS3 面板类、按钮字段和资源名
候选功能包（backpack1/Common1/EIcon/shop/...）
  ↓ 在 local-resources/regima/source/restored-swfs/ 中用 FFDec 窄查
SymbolClass / MovieClip / character id / 时间轴
  ↓ 只导出当前界面需要的图片、shape、sprite 或帧
local-resources/regima/task-outputs/ 批次输出
  ↓ 透明边缘、九宫格、字体、状态帧和动画验收
现代 stableKey / atlas / Phaser 布局
```

这里需要区分四种完成状态：

1. **源语料库已恢复**：知道 UI 可能在哪些 SWF 中；
2. **精确符号已定位**：知道具体 MovieClip、character id 和状态帧；
3. **派生素材已就绪**：PNG/SVG/atlas 已选择性导出并通过视觉检查；
4. **现代 UI 已接入**：stableKey、布局和交互进入 Phaser。

Enigma 突破解决了第一层，并让后三层成为可执行工作；它不等于所有 UI 已经自动拆分和接入。

## 5. Enigma 解包后为什么不能直接进 FFDec

加载代码表明，游戏读取某些资源包后会重新排列前 176 个字节：

```actionscript
output.writeBytes(input, 97, 79);
output.writeBytes(input, 0, 97);
output.writeBytes(input, 176);
```

也就是把输入数据转换为：

```text
input[97..175] + input[0..96] + input[176..end]
```

这是一种轻量文件头混淆。即使 Enigma 成功释放出 `assets/WuKong.swf`，它的开头也可能不是标准 `FWS/CWS/ZWS`，FFDec 因而仍会拒绝打开。

本项目已验证的正确流程是：

```text
原 EXE
  ↓ Enigma 虚拟文件系统解包
assets/WuKong.swf（可能仍有头部混淆）
  ↓ 按 AS3 逻辑重排前 176 bytes
标准或可识别的 SWF
  ↓ FFDec
MovieClip / Sprite / Shape / Bitmap / SymbolClass / AS3
```

这也解释了为什么“有 FFDec”不等于“现在就能取得所有素材”：FFDec 处理的是 SWF 格式，不负责先还原 Enigma 容器和游戏自定义的字节变换。

## 6. 能否解决问题：已知与待验证

| 问题 | 当前结论 |
| --- | --- |
| EXE 是否使用 Enigma 容器 | 已确认，存在巨大的 `.enigma1` 和 `.enigma2` 节 |
| EXE 是否记录目标资源路径 | 已确认，能找到 `WuKong.swf`、`level11.swf` 等完整路径 |
| AS3 是否知道资源用途 | 大量目标已经知道，且可达到较高语义置信度 |
| 目标文件的数据是否真的在容器中 | 已确认；恢复出角色、怪物、关卡等原始命名包 |
| `evbunpack` 是否支持这种具体 Enigma 版本 | 已确认；0.2.6 完整提取成功且复提取哈希一致 |
| 解包文件是否可直接被 FFDec 打开 | 多数需执行 176-byte 重排；174 个还原后 SWF 可用 |
| 所有动画是否能完整导出 | 代表样本已通过；后续按资源族选择性验证和导出 |

因此，Enigma 路线已经完成从假设、最小实验到全量恢复的验证；后续工作是按具体资源族定位和接入，而不是继续争论容器中是否存在素材。

## 7. 推荐的最小风险实验

不要一开始就覆盖项目文件，也不要运行全量 FFDec 帧导出。推荐分阶段验证。

### 阶段 A：只列虚拟文件目录

工具准备好后，先使用只列目录模式：

```powershell
evbunpack -l "D:\web-projects\Front-end\再续天庭-现代化\再续天庭1.1.exe" "D:\web-projects\Front-end\再续天庭-现代化\local-resources\regima\source\unpacked"
```

本阶段的成功标准：输出目录表中出现 `assets/WuKong.swf` 和 `assets/levels/level11.swf`。

`-l` 的具体输出行为应以所下载版本的 `evbunpack --help` 为准。

### 阶段 B：解到 Git 忽略的本地 RegiMA 目录

确认目录表正确后，再解包到项目内但由 Git 忽略的 RegiMA 资源根，例如：

```powershell
evbunpack "D:\web-projects\Front-end\再续天庭-现代化\再续天庭1.1.exe" "D:\web-projects\Front-end\再续天庭-现代化\local-resources\regima\source\unpacked"
```

不要把输出直接写入 `local-resources/regima/legacy-extraction/`。现有提取结果是逆向证据，应保留，方便比较旧方法和新方法。

### 阶段 C：先验证三个代表样本

建议只选：

1. `assets/WuKong.swf`：角色动画代表；
2. `assets/levels/level11.swf`：关卡场景代表；
3. `assets/Monster1111.swf`：怪物资源代表。

对每个文件记录：

- 解包后的实际路径和大小；
- 前 16 bytes；
- 是否已经以 `FWS/CWS/ZWS` 开头；
- 执行 176-byte 重排后是否成为有效 SWF；
- FFDec `-dumpSWF` 是否能读到合理标签；
- 是否存在预期 SymbolClass 或导出类名。

### 阶段 D：验收后再全量处理

只有三个代表样本通过，才值得：

- 建立 170 个资源包的正式清单；
- 选择性调用 FFDec 导出图片、形状、Sprite 和脚本；
- 将标注更新为 `source-corpus-ready`，精确定位后再推进为 `export-ready`；
- 开始角色帧、特效帧和关卡素材的正式拆分。

## 8. 如何判断实验到底成功没有

仅仅看到“解包完成”还不够。至少应满足以下条件：

1. 路径正确：还原出 `assets/...` 目录，而不是又得到一批数字文件；
2. 文件大小合理：不是 0 bytes 或几十 bytes 的假文件；
3. 结构合理：重排后存在正常 SWF 头、版本、声明长度和标签序列；
4. 语义对应：`WuKong.swf` 中能找到角色相关类或时间轴，`level11.swf` 中能找到关卡对象；
5. 可视结果正确：FFDec 导出的若干帧与原游戏录像或截图一致；
6. 可复现：相同命令再次运行得到相同文件和哈希。

如果目录能列出但文件无法解压，问题通常是解包器版本不兼容。如果文件能释放但 FFDec 不能打开，应先检查 176-byte 重排，而不是立即判定资源损坏。如果结构可以解析但没有预期角色，则再检查是否加载了 `SpecialUI/WuKong.swf` 等变体。

## 9. 标注工程应如何理解这次发现

标注工程已经根据 Enigma 结果升级为分阶段台账：

```text
source-corpus-ready
  ↓ 定位精确源包和符号
export-ready
  ↓ 选择性导出并验收
derived-ready
  ↓ 注册 stableKey、atlas 和布局
ready
```

`missing-original` 现在只用于“检索恢复语料库后仍确认缺少源文件”的少数情况，不能再用来表示“源 SWF 已有但尚未导出”。详细规则见 [`asset-annotation/annotation-schema.md`](asset-annotation/annotation-schema.md)。

## 10. 安全与边界

- 只逆向自己拥有或获授权分析的软件和素材；
- 原 EXE 保持只读，先记录 SHA-256；
- 解包结果放在项目外的新目录；
- 不直接运行来源不明的解包器，先扫描并核对来源；
- 不覆盖 `local-resources/regima/legacy-extraction/`；
- 不把“工具报告成功”当成素材恢复成功，必须做结构和视觉验收；
- 首次实验只处理文件目录和三个代表样本，避免产生数十 GB 的无效帧导出。

## 11. 一句话理解整个问题

AS3 像“资源目录和使用说明书”，Enigma 像“把仓库封进 EXE 的集装箱”，FFDec 像“打开每一个 SWF 箱子的工具”。现在集装箱已经解开并验证；接下来要按一个个具体 UI 和资源族，从正确箱子中选择性取出素材并接入现代工程。
