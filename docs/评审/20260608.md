二、PetSystem.ts 深度评审：质量好，但膨胀速度已失控
做得好的
技能函数签名一致性好。 16 个 requestPet*Skill() 函数全部遵循同一模式：


(params: { roster, runtime, targets, projectiles, random? }) => PetSkillCastResult
每个函数有相同的门禁检查顺序：无出战宠物 → 物种/形态不对 → 未学技能 → MP不足 → CD未好 → 无目标 → 距离检查。这种一致性使 review 和调试变得简单。

自动 buff 系统设计完整。 6 个 buff（sxkb/fsnl/smjc/mfjc/gjjc/fyjc）每个都有独立的触发条件、counter 倒计时、buff 持续时间和失效回滚。updateActivePetAutoBuffEffect 在 buff 到期时正确回滚了主人属性（包括 HP/MP 按比例回退）。

经验曲线有明确的数值模型。 getPetExperienceToNextLevel 使用分段公式（L1-10 线性，L11+ 二次增长），PetHpByLevel 和 PetDefenseByLevel 以 90 级数组存储。getPetBaseStats 使用物种基础值 + 品质修正的公式。

形态进化处理得当。 updatePetFormForLevel 在升级时自动检测形态转换（L16→F2, L30→F3），形态变化被记录为 PetFormChange 事件。

问题
1. 代码重复极其严重。 16 个 requestPet*Skill() 函数中，每个函数的前 5-6 个门禁检查几乎完全相同（约 15 行模板代码重复 16 次 = ~240 行重复）。这些可以抽取为通用门禁管线：


// 当前模式（重复16次）:
const pet = getActivePet(params.roster);
if (!pet) return setPetSkillFailure(...);
const state = ensurePetSkillState(pet);
if (pet.species !== 'xxx' || pet.form !== N) return ...;
if (!pet.skills.includes('skillName')) return ...;
if (pet.mp < cost) return ...;
if (state.xxx.cooldownMs > 0) return ...;
建议抽取为泛型门禁函数：


function validatePetSkillGate<T extends keyof PetSkillState>(
  params, species, form, skillKey, mpCost, cooldownKey
): { pet, state } | PetSkillCastResult { ... }
2. updatePetSkillState 中有 16 行完全重复的 Math.max(0, ...cooldownMs - deltaMs)。 PetSystem.ts:1012-1028 每行只有 cooldown 字段名不同。可以用一个数组遍历或 helper 替代。

3. updatePetAutoBuffs 中 updateActivePetAutoBuffEffect 被调用 6 次，每次模式相同。 PetSystem.ts:1050-1067 这 6 行重复可以用循环或 map 替代。

4. 6 个 tryTriggerPet*AutoBuff 函数结构完全相同。 PetSystem.ts:2987-3287 每个函数都是：检查技能存在 → 检查 counter/active → 检查 MP → 计算加成值 → 设置 active → 重置 counter。300行代码中约 70% 可抽象。

5. createSeed*Pet 四个函数完全重复。 createSeedMonkeyPet、createSeedHorsePet、createSeedDragonPet、createSeedTurtlePet 在 PetSystem.ts:551-701 中只有 species 参数不同（~150 行重复）。应为 createSeedPet(species, params) 的单一实现。

6. 文件已达 3,554 行。 这违反单一职责原则。建议最小拆分：

PetState.ts — 类型定义 + 数值表（PetTuning、PetHpByLevel、PetDefenseByLevel）
PetSkillSystem.ts — 16 个技能释放函数
PetAutoBuffSystem.ts — 6 个自动 buff 系统
PetProgressionSystem.ts — 经验、升级、形态进化、技能学习
PetSystem.ts — 核心（roster、runtime、捕捉、消耗品）
三、ProjectileSystem.ts 评审：扩展有序但碎片化
做得好的
Tuning 常量化模式统一。 24 个新 projectile tuning 全部使用 as const 对象，类型安全且不可变。spawnProjectileFromTuning 的泛型参数正确地覆盖了所有变体类型。

ProjectileModel 扩展向后兼容。 新增字段全部使用可选属性（magicStunMs?、magicIceMs?、trackingTargetId?、petHealOnHit? 等），不影响已有 Role2 技能 projectile 的逻辑。

问题
1. spawnProjectileFromTuning 的 union type 过长。 ProjectileSystem.ts:1237-1265 — 28 个 typeof ...Tuning 的 union。每次新增 projectile 类型需要同时更新 3 处：ProjectileVariant union、spawnProjectileFromTuning union、spawnXxxProjectile 函数。

2. spawn 函数有 20+ 个极为相似的包装函数。 每个 spawnPet*Projectile 函数都是：spawnProjectileFromTuning → 覆盖 damage → 设置 destroyWhenSourceHurt=false → push。6行代码重复 20 次 = 120 行重复。这是合理的"薄包装"还是过度拆分？我认为薄包装合理，但可以模板化。

3. ProjectileModel 字段膨胀。 从 36 个字段增长到 ~50 个字段。若干字段只在特定变体中有意义（petHealOnHit 只在龙系宠物技能中有意义，magicStunMs 只在法宝 projectile 中有意义）。这是"宽模型"vs"窄模型"的经典取舍——当前选择宽模型是可以接受的，但文档应说明哪些字段对哪些变体有效。

八、当前最紧迫的三个问题
1. PetSystem.ts 拆分（紧迫度：高）
3,554 行的单一文件在每次修改时都牵一发而动全身。建议的拆分方案（至少拆成 4-5 个文件）在上面已经给出。

2. 16 个技能函数中的重复代码消除（紧迫度：中）
这不是架构问题，是代码质量问题。可以在不改变任何外部 API 的情况下，通过提取门禁函数减少 ~300 行重复代码。

3. TestScene 拆分（紧迫度：中）
仍然是最关键的中期风险。每次新增系统都在增长 TestScene。建议至少先把视图工厂函数（createMonsterView、createBossView、createPetView 等 ~600 行）移到渲染 helper 模块。

九、一句话总结
一周内以极高的测试覆盖率完成了宠物全技能链和法宝大半系统，工程推进速度令人印象深刻；但 PetSystem.ts 已达 3,554 行且包含大量可消除的重复代码，若不及时拆分，它将成为下一个 TestScene 级别的维护负担。建议下一轮优先做 PetSystem 的模块拆分和重复代码消除，再继续扩展新功能。
