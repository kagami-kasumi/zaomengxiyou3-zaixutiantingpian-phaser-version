# 怪物与关卡组织中的设计模式

本文说明现代版为什么需要重构怪物与关卡的组织方式、该方向与面向对象理念的贴合度，以及它和 Java 传统设计模式的联系。本文讨论现代 TypeScript/Phaser 架构，不要求照搬原版 AS3 类结构，也不把设计模式当作必须套用的类模板。

## 1. 为什么需要重构

当前工程已经具备若干正确边界：

- `Stage*Layout` 保存关卡平台、停点、刷怪点、机关和传送门等内容数据。
- `Stage*FlowSystem` 管理波次、同屏上限、Boss、失败和通关。
- `Stage1CombatSystem` 实际被多个正式关卡复用，提供怪物数值、索敌、接近、攻击阶段和伤害结算。
- `MonsterPhysicsSystem`、`MonsterDefeatRewardSystem` 和 `DropSystem` 已分别拥有怪物物理、死亡奖励和掉落规则。
- 怪物视觉系统与 Phaser bridge 已能跨相邻关卡复用部分怪物素材和动作。

但当前结构仍有四个扩展风险：

1. **通用能力带有关卡名称。** `Stage1CombatSystem` 已经是跨关卡 owner，却仍以首个切片命名，导致“通用规则”和“关卡适配”在语义上混杂。
2. **同一怪物存在两个运行时登记点。** 部分关卡 Flow 保存 `aliveEnemies`，场景 bridge 又保存带战斗、物理和视图的怪物 `Map`，必须靠人工同步创建、死亡和移除。
3. **特殊怪物重复实现共同算法。** `Monster3System` 与 `Monster30System` 分别拥有最近目标选择、状态转换和攻击决策；继续按怪物复制会增加修复和校准成本。
4. **内容差异与行为差异没有稳定接缝。** 数值、物理类型、AI、技能、动画集和掉落表需要组合，但目前部分由通用表拥有、部分由专属系统拥有、部分在关卡 bridge 中连接。

重构目标不是“让类更漂亮”，而是建立唯一的数据与生命周期 owner，使以下变化可以局部完成：

- 同一种怪物出现在不同关卡时，只复用怪物定义和行为，不复制实现。
- 多种地面近战怪共用索敌、接近和攻击决策，只替换数值、动作和技能。
- 飞行、远程、施法和 Boss 可以替换行为策略，而不扩张出深层继承树。
- 关卡只决定何时、何地、生成什么，以及何时算遭遇完成；不实现怪物内部战斗规则。
- Phaser 视图只消费怪物状态并播放视觉，不成为战斗事实来源。

## 2. 推荐的职责边界

### 2.1 关卡拥有遭遇，不拥有怪物内部规则

关卡应拥有：

- 地形、平台、障碍、机关和出口；
- 出生点、波次、生成延迟和同屏上限；
- 停点、封路、Boss 触发和通关条件；
- 本次遭遇关联的怪物稳定 ID 或存活计数。

关卡不应拥有：

- 怪物如何选择玩家；
- 怪物如何接近、攻击、受击和死亡；
- 怪物基础数值、状态抗性和技能内部算法；
- 动画帧推进、命中结算和掉落随机算法。

“关卡只负责怪物列表”仍然过窄。准确说法应是：**关卡负责遭遇编排，怪物系统负责怪物运行。**

### 2.2 怪物静态定义与运行状态分离

推荐区分：

```ts
type MonsterDefinition = Readonly<{
  monsterTypeId: string;
  statsProfileId: string;
  physicsProfileId: string;
  brainId: string;
  abilitySetId: string;
  animationSetId: string;
  rewardProfileId: string;
  lootTableId: string;
}>;

type MonsterRuntime = {
  id: string;
  monsterTypeId: string;
  x: number;
  y: number;
  hp: number;
  state: string;
  targetId?: string;
};
```

`MonsterDefinition` 是可复用内容配置；`MonsterRuntime` 是一只具体怪物在本局中的可变状态。关卡出生点只引用 `monsterTypeId`，由怪物工厂完成定义解析与运行时创建。

### 2.3 唯一运行时注册表

`MonsterRuntimeRegistry` 应成为存活怪物的唯一登记点，第一版只需要：

- 稳定 ID；
- 创建、查询和安全移除；
- 按怪物类型或遭遇 ID 查询；
- 幂等死亡/移除；
- 向关卡 Flow 返回生命周期结果。

关卡 Flow 可以保存遭遇 ID、生成计划、已生成数量和已击败数量，但不再复制完整怪物状态。第一版不引入完整 ECS、全局事件总线或依赖注入容器。

### 2.4 行为使用组合，而不是深继承

推荐把共同 AI 表达成策略：

```ts
type MonsterBrainId =
  | 'ground-melee'
  | 'ground-ranged'
  | 'flying-melee'
  | 'flying-ranged'
  | 'stationary-caster'
  | 'scripted-boss';
```

地面近战策略可统一完成：

1. 从合法存活目标中选取目标；
2. 判断警戒范围；
3. 面向目标；
4. 未进入攻击距离时移动；
5. 进入距离后进入攻击状态机；
6. 攻击结束后回到追踪状态。

怪物之间的差异由参数和能力集合表达。Boss 的阶段脚本可以是独立策略，但仍复用目标选择、伤害、物理、状态效果和死亡结算。

### 2.5 动画和掉落不由怪物对象直接执行

怪物运行状态只表达 `approach/windup/active/recovery/hurt/dead` 等事实。动画系统根据状态和 `animationSetId` 选择素材、帧和播放规则；Phaser bridge 负责显示对象。

同理，怪物定义只引用 `lootTableId` 和 `rewardProfileId`。死亡奖励系统负责随机、生成、归属和拾取，不应让每个怪物子类自行创建掉落物。

## 3. 与面向对象理念的贴合度

### 3.1 封装

封装不是“把字段放进 class 并设为 private”，而是让规则有明确 owner：

- 目标选择规则属于目标选择/AI owner；
- 重力和落地属于怪物物理 owner；
- 伤害与命中去重属于战斗 owner；
- 波次与通关属于关卡 Flow；
- 动画播放属于视觉 owner。

调用者只依赖公开合同，不直接修改另一个系统的内部状态。

### 3.2 抽象

抽象应来自稳定变化轴，而不是为了减少文件数：

- 所有怪物都有稳定 ID、类型、位置、生命和生命周期，这是稳定抽象。
- “地面近战”“飞行远程”是可复用行为族，可以抽成策略。
- Monster16 的具体六种攻击不是所有怪物共同能力，不应塞进基础怪物抽象。

### 3.3 多态

Java 常用接口实例实现多态；TypeScript 中也可以用判别联合、函数表或满足同一接口的纯函数实现：

```ts
type MonsterBrain = {
  update(context: MonsterBrainContext): MonsterBrainResult;
};
```

多态的价值是调用方只知道 `brain.update()`，而不是要求所有实现必须继承同一个有状态父类。

### 3.4 继承与里氏替换

所有怪物在领域上都可以称为 `Monster`，但不代表每个怪物都应该继承一个包含 AI、动画、掉落和物理的巨大 `Monster` 基类。

如果父类假设“怪物受重力、沿 X 轴接近、只有一个攻击窗口”，飞行怪、固定炮台和多阶段 Boss 会不断覆盖父类方法，最终违反里氏替换原则：调用方无法再把任意子类当作具有一致语义的父类使用。

因此现代实现只保留最小共同合同，差异通过组合获得。继承可以用于非常稳定、确实满足替换关系的局部视图适配，但不作为怪物玩法主架构。

### 3.5 SOLID 对应关系

- **SRP 单一职责：** Flow 编排遭遇，Brain 决策，Physics 移动，Combat 结算，Visual 播放，Reward 产出奖励。
- **OCP 开闭原则：** 新增怪物通常增加定义或策略，不修改每个关卡和通用调度循环。
- **LSP 里氏替换：** 所有 Brain 遵守相同输入/输出合同，不依赖脆弱父类默认行为。
- **ISP 接口隔离：** 飞行怪不必实现地面落地接口；无掉落单位不必伪造掉落逻辑。
- **DIP 依赖倒置：** Flow 依赖“生成怪物/收到死亡结果”的合同，不依赖 Phaser Sprite 或具体 Monster16 类。

## 4. 与 Java 传统设计模式的联系

| 经典模式 | 本项目中的对应 | 适用理由 | 注意事项 |
| --- | --- | --- | --- |
| Strategy 策略 | `MonsterBrain`、目标选择、移动和攻击选择策略 | 同一运行时可以替换地面近战、飞行远程或 Boss 决策 | 不要把每个数值差异都变成一个策略类 |
| State 状态 | `approach/windup/active/recovery/hurt/dead` 状态机 | 明确动作转换、不可重入窗口和死亡生命周期 | 简单状态可用联合类型与纯函数，不必为每个状态创建 class |
| Factory Method / Abstract Factory 工厂 | 根据 `monsterTypeId` 创建定义、运行状态、物理与视觉适配 | 隔离关卡出生数据和具体构造细节 | 工厂不应同时承担更新、查询和销毁 |
| Registry 注册表 | `MonsterDefinitionCatalog`、`MonsterRuntimeRegistry` | 提供定义查找和唯一运行时 owner | 静态定义目录与可变运行时注册表必须分开 |
| Adapter 适配器 | `Stage*GameplayBridge`、怪物视觉 bridge | 把 Phaser 对象适配为系统快照和命令 | Adapter 中不得重新实现领域规则 |
| Facade 外观 | 对场景暴露少量怪物运行时调度入口 | 降低场景对多个底层系统的直接耦合 | Facade 只编排，不囤积规则和状态 |
| Observer / Domain Event 观察者 | 怪物生成、受击、死亡、移除结果 | 让 Flow、奖励和视觉响应同一生命周期事实 | 第一版优先显式返回事件数组，不急于引入全局事件总线 |
| Template Method 模板方法 | 传统 `BaseMonster.update()` 的固定流程 | 可作为理解共同生命周期的启发 | 在 TypeScript 中优先组合纯函数，避免父类 hook 膨胀 |
| Flyweight 享元 | 多只同类型怪物共享只读 `MonsterDefinition` 和动画元数据 | 避免每个实例复制静态表 | 运行时 HP、目标、状态绝不能共享 |
| Command 命令 | AI 输出移动、开始攻击、释放技能等意图 | 决策与执行分离，便于确定性测试 | 不必为每条命令创建 Java 式类层级 |

这些模式是对变化关系的命名，不是代码生成模板。Java 中常以接口、抽象类和对象实例表达；TypeScript 项目可以用类型、只读配置、函数映射和纯函数获得相同解耦效果。

## 5. 不采用的方向

### 5.1 不建立万能 `BaseMonster`

禁止把数值、AI、物理、动画、掉落、状态效果和 Phaser Sprite 全部塞入一个父类。它会形成 God Object，并迫使特殊怪物覆盖大量不适用行为。

### 5.2 不一次性引入完整 ECS

项目目前需要的是唯一稳定 ID 和生命周期登记，而不是 archetype、稀疏组件数组、系统调度图和复杂查询语言。只有实体数量、跨系统查询和性能证据确实要求时，才继续演化。

### 5.3 不为“复用”抹平原版差异

共同策略必须允许参数化警戒距离、二维/横向距离、攻击窗口、技能选择和移动方式。不能为了共用一段代码，把飞行怪、Boss 或原版特殊动作强制改成普通地面怪行为。

### 5.4 不让关卡 Flow 成为实体仓库

Flow 需要知道遭遇进度，但不需要持有战斗、物理、动画和掉落的完整副本。通关判断应消费注册表查询或明确的生命周期结果。

## 6. 推荐迁移顺序

1. 建立 `MonsterDefinition`、`MonsterRuntime`、`MonsterBrain` 和目标选择合同，用兼容 facade 保持现有关卡行为。
2. 把 `Stage1CombatSystem` 中真正通用的怪物表、创建和更新逻辑迁移到无关卡命名的 owner。
3. 将 `Monster3System`、`Monster30System` 的共同目标选择接到统一策略，保留其专属攻击和状态效果。
4. 建立最小 `MonsterRuntimeRegistry`，选择一个同时具有普通怪和 Boss 的正式关卡做试点。
5. 试点通过后，再按关卡逐批消除 Flow 与 scene bridge 的双重怪物登记。
6. 最后统一动画状态映射与工厂入口；每批保持原行为、系统测试、构建和运行验收通过。

迁移过程中应保持旧 facade 可用，先建立新 owner，再迁移消费者，最后删除重复实现。不得在一次 task 中同时迁移全部正式关卡、全部特殊怪物和全部视觉 bridge。

## 7. 完成判断

怪物架构重构只有在以下条件满足时才能关闭：

- 同一种怪物跨关卡只引用一个静态定义 owner；
- 存活怪物只有一个权威运行时注册表；
- Flow 不再复制完整怪物运行状态；
- 至少一个普通行为族复用统一 Brain/Targeting；
- 特殊怪物通过组合或策略表达差异，没有扩大万能父类；
- 动画、伤害、物理、奖励和掉落 owner 边界明确；
- 已迁移正式关卡的确定性测试、构建与运行行为无回归。
