# PG-001 共享技能规则重复定义

状态：方案试行中。

来源：`docs/评审/five-roles-review-2025-06-26.md` 的 CH1-CH3，以及用户确认“重复定义”才是问题；“数值校准需要跨文件同步、容易遗漏”只是后果。

## 1. 问题定义

五角色技能系统中存在共享规则重复定义：

- 共享 MP 表、伤害公式和段数表在多个技能系统中重复定义。
- `clampLevel` / `findJustPressedSkillSlot` 等基础 helper 在多个文件中独立实现，且签名和行为细节不一致。

问题不是“调数值麻烦”，而是同一条共享规则存在多个现代定义源。数值校准需要跨文件同步、容易遗漏，是重复定义带来的影响，不是问题本体。

`HeroSkillModel` runtime 全量承载属于另一个耦合问题，不并入 PG-001；它可作为后续独立问题治理记录。

## 2. 证据

- 评审记录指出 `consumeMpByLevel` 在 15+ 个文件中逐字重复。
- `skillFactorBase` / `skillFactorPerLevel` / `fixedDamageCount` 在多个角色技能系统中重复定义。
- `clampLevel`、`clampLevelOrZero`、`findJustPressedSkillSlot` 存在多个局部实现。

## 3. 解决方案

解决方案不是单次抽取代码，而是一条约束 AI 生成代码的 harness 规则：**共享定义所有权约束**。

AI 在新增或修改代码时，如果准备定义满足以下条件之一的内容，必须先定位现有 owner，而不是直接复制到当前文件：

- 跨角色、跨技能、跨系统通用的数据表，例如 MP 表、通用伤害段数表、通用技能系数。
- 跨文件可复用的纯函数 helper，例如等级裁剪、技能槽输入检测、距离计算。
- 同一业务规则的第二份现代定义。

执行规则：

1. 先用 `rg` 搜索稳定短关键词，确认是否已有 owner。
2. 如果已有 owner，必须复用或扩展 owner，不得在新文件重复定义。
3. 如果没有 owner，但预计会有两个及以上消费者，应先创建 owner 模块，再让消费者引用。
4. 如果只服务单个技能或单个角色，可保留局部定义，但注释或命名应体现其局部性。
5. 评审时发现新增重复定义，应退回修改；不能只靠“以后再抽”通过。

当前候选 owner：

- `SkillTuning.ts`：共享 MP 表、通用伤害段数表、通用技能系数。
- `SkillInputUtils.ts`：技能槽输入检测。
- `SkillMathUtils.ts`：等级裁剪、通用数学 helper。

这是一条生成约束，不是一次性重构任务。当前代码抽取只能作为首个样本，不能单独证明 harness 方案可行。

## 4. 测试方案

本问题测试的是 harness 约束是否能影响未来 AI 生成代码，而不是只测试当前代码是否能跑。

短期检查：

- 使用 `rg` 搜索旧重复模式，例如局部 `consumeMpByLevel`、局部 `findJustPressedSkillSlot`、重复 `skillFactorBase` / `skillFactorPerLevel`。
- 运行 `npm run check:workflow`，确认问题治理规则仍被脚手架接住。

长期可行性验证：

- 后续至少 3 次涉及技能、数值、helper 或跨角色共享规则的代码生成任务，都必须在收尾时检查是否遵守“共享定义所有权约束”。
- 每次样本记录任务、日期、涉及文件、是否新增重复定义、验证命令或搜索关键词。
- 3 次样本全部没有新增重复定义，才说明该 harness 方案初步可行。
- 任意一次样本新增重复定义，则判定当前方案不可行或不充分，需要升级方案，例如加入更强的校验脚本、模板约束或评审必检项。

## 5. 测试结果

当前结果：

- 已完成一次历史样本：五角色技能评审整改中抽取了 `SkillTuning.ts`、`SkillInputUtils.ts` 和 `SkillMathUtils.ts`，并通过 `npm run check:structure`、`npm run test:systems` 和 `npm run build`。
- 该样本只能证明“旧重复定义可以被修复”，不能证明“harness 能约束未来生成代码不再重复定义”。
- 2026-07-18 针对工程评审的复核中，当前工作树搜索没有发现局部 `consumeMpByLevel`、`skillFactorBase`、`skillFactorPerLevel`、`fixedDamageCount`、`clampLevel` 或 `findJustPressedSkillSlot` 定义；现有消费者引用 `SkillTuning.ts`、`SkillInputUtils.ts` 和 `SkillMathUtils.ts`。这说明旧问题在当前快照中没有回归。
- 文件历史显示 owner 抽取完成于 2026-06-29；此后没有涉及这些共享技能规则的代码生成提交，因此仍没有符合“后续相关代码生成任务”定义的观察样本。2026-07-18 的评审复核不是代码生成任务，不计入下表 3 个样本。
- 同日评审曾把 2026-06-26 的旧发现直接写成“全部未修复”。这不是共享定义所有权约束失效，而是评审没有复核当前证据。已在 `review-protocol.md` 增加证据时效与遗留发现复核规则。

阶段判断：当前方案对“清理既有重复并保持当前快照单一 owner”有效；对“约束未来 AI 代码生成不再复发”尚无足够样本，PG-001 继续保持“方案试行中”，不提前关闭，也不因一次错误评审升级为自动检测脚本。

## 6. 关闭标准

满足以下条件时，PG-001 才可视为关闭：

- 问题定义稳定为“重复定义”，不把影响误写成问题本体。
- `code-quality-gates.md` 或同级脚手架文档明确声明共享定义所有权约束。
- 后续至少 3 次相关代码生成任务均未新增共享规则重复定义。
- 每次样本都有搜索或评审证据，而不是只看系统测试通过。
- 若 3 次样本中出现新增重复定义，必须重新设计方案，不能关闭本问题。
