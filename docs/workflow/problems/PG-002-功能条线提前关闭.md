# PG-002 功能条线被局部切片提前关闭

状态：治理规则已落地，方案补强中。

来源：2026-07-16 用户对“炼丹炉条线已全部收束”的纠正，以及 2026-07-18 对后续任务组织方式的连续澄清。

## 1. 问题定义

当前任务体系把机制、task 和纵向切片作为主要追踪层级，却没有把“完整玩家系统”定义为持续持有的承诺单位，也没有规定同一功能线必须连续执行。

第一阶段缺少大量真 UI 和原版视觉资源，按功能切片验证输入、战斗、背包、合成等机制是否可实现，是与当时资源条件相适应的策略。RegiMA 恢复语料库出现后，项目已经有条件交付真 UI、完整权威内容和正式可达流程；任务组织方式却仍停留在“完成一个局部切片后即可选择其他 Ready task”。

这会产生两类系统性错误：

- 把代表性切片完成越级解释为整条功能线完成。
- 即使承认条线未完成，也可能在局部 task 归档后跳转到其他系统，让未完成系统长期悬置并反复返工。

炼丹炉案例中，`VS-042..044` 证明了合成事务和两个真图标配方族可以工作，但用户要求的交付物是：玩家正式可见、使用真 UI、覆盖权威合成表中所有合成的炼丹炉系统。前者不能替代后者。

问题本体因此不仅是“缺少功能条线台账”，而是当前 harness 缺少以下统一约束：

- 完整系统的范围所有权。
- task 对完整系统的从属关系。
- 严格单条功能线 `WIP=1`。
- 阻塞时继续治理当前条线、禁止切线的调度规则。
- 基于全量覆盖证据的关闭门禁。

若不治理，关卡、装备、角色、UI、资源恢复和其他内容密集系统都会重复出现“做了很多切片，却没有一个系统真正结束”的问题。

## 2. 证据

- `docs/workflow/task-generation.md` 只定义 `TASK-SETTINGS`、`TASK-ARCH`、`TASK-SLICE`，没有完整系统/功能条线实体、激活状态、连续调度或关闭证据字段。
- `docs/workflow/agent-protocol.md` 的 `/goal` 默认粒度是一个 Ready task 或一个可交接产物，没有要求系统级目标跨多个 task 持续存在。
- `docs/tasks/vertical-slices.md` 的 `VS-042..044` 分别验证合成事务、首个视觉配方和第二个视觉配方；它们是有效局部闭环，不是全量交付。
- `docs/reverse-engineering/mechanics-index.md` 的 `M-039` 记录 67 个 `direct_static`、41 个默认继承和 4 个特殊继承唯一配方，共 112 个配方；既有真图标/UI 闭环只覆盖其中两个配方族。
- `docs/reverse-engineering/crafting-ui-index.md` 曾把强化、分解、制作页签排除在既有视觉切片之外，证明“容器可见”也不能自动代表完整系统。
- `docs/tasks/task-history.md` 的 `TASK-SLICE-118` 曾写出“炼丹炉条线已全部收束”，把 task 级事实外推为条线级结论。
- 资源标注状态从大量“缺来源”迁移为 `source-corpus-ready`，说明继续只做占位 demo 已不再是唯一可行路线。
- 用户明确确认：一个完整系统内部可以拆成多个 task，但 task 必须连续；全项目严格单线 `WIP=1`，遇到阻塞时解决阻塞，不切换系统。

## 3. 解决方案

### 3.1 任务层级

为现有机制、task 和纵向切片之上增加轻量的**功能条线**（`LINE-*`）：

- 功能条线是对用户作出的完整系统交付承诺。
- task 是实现该承诺的内部执行单位，可以保持小而可验证。
- 纵向切片是技术验证、阶段样本或回归证据，不是功能条线完成的充分条件。
- 一个功能条线可以跨多个 Goal、task 和对话，但在关闭前始终保有目标所有权。单个 Goal 只是一次 `/goal` 的有界交接包，默认一 task、预计零次 compact；具体规模由 PG-008 的主工作包和验收批次门禁约束。

建议新增 `docs/tasks/feature-lines.md`。每条记录至少包含：稳定 id、用户确认范围、明确排除项、覆盖矩阵、激活状态、关联 task、当前阻塞、下一 task 和关闭证据。

### 3.2 严格单线 WIP=1

项目在任意时刻最多只能有一个激活功能条线：

1. 选择一个完整系统并确认范围。
2. 激活对应 `LINE-*`，生成或选择该线的首个 task。
3. task 完成后归档；Goal 的绑定 task 全部完成后 Goal 关闭，但功能线保持激活。
4. 立即激活或生成同一功能线的下一 Goal/task，然后结束当次 `/goal` 交接；不得转向其他系统，也不在同一次 `/goal` 隐式续跑下一 Goal。
5. 若遇阻塞，登记阻塞证据，并在同一功能线中生成资源定位、逆向、人工验收或工程修复 task 来解除阻塞。
6. 阻塞期间不得激活或推进第二条功能线。
7. 只有覆盖矩阵和关闭门禁全部满足后才能关闭当前线；关闭后才允许选择下一条线。

这不是要求把完整系统塞进一个巨大 task。相反，task 仍需保持清晰边界，只是调度权服从当前功能线，而不是服从全局 Ready 列表。

### 3.3 完整系统关闭合同

功能线关闭标准必须根据系统定制，但至少检查以下维度：

- **真实呈现**：玩家看到的是目标真 UI 和所需原版视觉资源，不是仅供验证的 demo/占位界面。
- **权威内容全覆盖**：权威数据源中的全部已确认内容都有覆盖矩阵，不能用几个代表样本代替全集。
- **正式可达**：系统从正式游戏流程进入并能完成核心操作，不只存在于 `TestScene` 或调试入口。
- **跨系统集成**：输入、库存、资源、数值、存档、奖励等相关边界按该系统范围闭合。
- **验证闭合**：自动测试、构建、资源台账、必要人工试玩和视觉验收均有证据。
- **剩余项诚实**：所有排除项由用户明确确认；未确认缺口不得包装成“以后优化”。

“以后不需要再管该系统”应理解为：已知权威范围已经完整交付并可从开发主线关闭；未来全局架构变化、真实 bug 或用户新增范围仍可触发维护，但不能以预先留下 UI、内容或流程缺口的方式假装一次完成。

### 3.4 LINE-CRAFTING 首个落地

`LINE-CRAFTING` 的用户确认范围是：交付玩家可见、带真 UI、覆盖权威合成表中所有合成的炼丹炉系统。

执行时需要：

- 从 112 个权威唯一配方生成材料与产物全集。
- 逐项核对定义、库存类别、获得或测试入口、图标、暂存、预览、消耗、产物显示、属性继承和成功/失败结果。
- 接入正式玩家流程并保留真 UI。
- 为全部规则类别建立自动测试，并完成必要视觉/试玩验收。
- 保留 `VS-042..044` 和既有 task 的真实完成事实，但撤回由此推导出的条线关闭结论。

强化、分解和制作是否属于同一炼丹炉交付范围，需要在建立 `LINE-CRAFTING` 覆盖矩阵时由用户明确；Agent 不得自行通过排除它们来缩小“完整炼丹炉”的含义。

### 3.5 协议和自动门禁

执行治理时需要同步更新：

- `agent-protocol.md`：功能线跨 Goal 持续持有目标，单次 `/goal` 只执行一个 Goal，同时保持严格单线和阻塞不切线。
- `task-generation.md`：task 必须关联激活 `LINE-*`，下一 task 必须保持同线。
- `task-board.md`：展示当前唯一激活功能线和同线推荐 task。
- `feature-lines.md`：成为功能线范围、状态和关闭证据的权威台账。
- `tools/validate-workflow.mjs`：自动拒绝双激活、跨线推荐、无覆盖关闭和阻塞切线等状态。

治理边界：本问题只重构任务所有权、连续调度和关闭门禁；炼丹炉剩余内容由 `LINE-CRAFTING` 的连续游戏 task 实际完成。两者分别关闭。

## 4. 测试方案

### 工作流结构验证

- 建立一个激活中的 `LINE-CRAFTING` 样例并运行 `npm run check:workflow`。
- 增加负向样例，确认以下状态必须失败：同时激活两条线、当前推荐 task 不属于激活线、完成线仍有未完成覆盖项、完成线缺关闭证据。
- 增加阻塞样例：激活线阻塞后若推荐另一功能线 task，校验必须失败；推荐同线的阻塞解除 task 应通过。
- 检查 task 归档后系统级 goal 和激活线仍保持，直到条线关闭。

### LINE-CRAFTING 回归验证

- 权威配方统计必须与 67 + 41 + 4 = 112 一致。
- 生成材料、产物、图标、正式获取/测试入口和 UI 覆盖集合，并检查集合差异。
- 既有两个视觉配方专项测试继续通过，证明条线恢复进行中不会否定已完成切片。
- 新生成炼丹炉 task 全部关联 `LINE-CRAFTING`，且完成一个 task 后推荐项仍属于该线。
- 在真 UI 和正式流程中执行代表各种规则类别的配方，并保存视觉验收记录。

### 连续调度样本

- 完整观察 `LINE-CRAFTING` 从激活到关闭的 task 序列，确认中途没有切换到其他功能线。
- 遇到至少一个真实或模拟阻塞时，确认下一步是同线解除阻塞，而不是换线。
- 后续再观察至少一个功能条线，确认相同规则不依赖炼丹炉特例。

## 5. 测试结果

当前结果：

- 已确认问题跨越任务生成、`/goal`、看板、历史、纵向切片、机制表和资源流程，不是单句 typo。
- 已确认 112 个唯一配方与两个完整真图标配方族之间存在明显覆盖差距。
- 用户已确认完整系统是承诺单位、task 可以拆分但必须连续、全项目严格单线 `WIP=1`、阻塞时不切线。
- 已建立 `feature-lines.md` 和 `feature-line-coverage/LINE-CRAFTING.md`，将 `LINE-CRAFTING` 设为唯一 `Active`，将 Stage 1-1 降为 `Planned`。
- 已把 `/goal`、正式 task、任务生成和看板统一为“功能线是承诺单位、task 是执行单位”；task 完成后继续同线下一 task，阻塞时不得切线。
- 已将 `TASK-SETTINGS-047` 设为合成线首个覆盖盘点 task；`TASK-SETTINGS-046` 在合成线关闭前不能进入 Ready。
- 已修正 `TASK-SLICE-118`、`M-039` 和资源项目状态中的整线完成外推，同时保留 `VS-042..044` 的真实完成事实。
- `validate-workflow.mjs` 已检查唯一 Active、task 功能线字段、跨线 Ready/Blocked、跨线推荐、Active 当前 task、Done 线遗留 task 和覆盖台账，并内置双 Active、跨线 Ready、跨线推荐、Done 遗留 task、阻塞切线五类负向样例。
- `npm run check:workflow` 通过：当前 2 个未完成 task、唯一推荐 `TASK-SETTINGS-047`、172 条资源标注均通过。

剩余观察：

- `LINE-CRAFTING` 尚未完成 112 配方逐项盘点和后续连续实现，因此无法验证完整线从激活到关闭的真实运行样本。
- 尚未观察下一条功能线从 Planned 切换为 Active 的第二个样本；PG-002 继续保持打开，不提前关闭。

## 6. 关闭标准

PG-002 只有同时满足以下条件才可关闭：

- `feature-lines.md` 已建立，并且任意时刻最多一个 `LINE-*` 处于激活状态。
- `agent-protocol.md` 和 `task-generation.md` 已明确“功能线是承诺单位、task 是执行单位”，并规定 task 连续推进、阻塞不切线。
- `task-board.md` 的当前推荐只能属于激活 Goal 和激活线；Goal 关闭不会隐式结束功能线。
- `validate-workflow.mjs` 能拒绝双激活、跨线推荐、阻塞切线、无覆盖证据关闭和完成线仍有未完成 task。
- `LINE-CRAFTING` 已按用户确认范围建立全量覆盖矩阵；既有切片保持完成，条线在全量交付前保持激活。
- 已观察 `LINE-CRAFTING` 的完整连续 task 序列，以及至少一个后续功能线样本，未再次发生切片越级关闭或中途切线。

以下情况只能算部分缓解，不能关闭：

- 只改掉“炼丹炉条线已完成”的文案。
- 只新增功能线台账，但仍允许多个激活线或任意切换 Ready task。
- 只把整个炼丹炉写成一个巨大 task，没有连续小 task 和覆盖矩阵。
- 只补协议文字，没有自动负向校验。
- 遇到阻塞时把当前线挂起并开发其他系统。
- 真 UI、全部权威内容或正式流程仍留作“后续补齐”，却宣告功能线完成。
