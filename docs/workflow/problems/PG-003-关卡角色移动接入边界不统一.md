# PG-003 关卡角色移动接入边界不统一

状态：已确认，待治理。

来源：2026-07-21 Stage 1-2 运行时校正中，用户指出角色移动不应随关卡重复实现；P1 K 在 Stage 1-2 未接入跳跃的缺陷证明现有边界尚未封闭。用户已明确确认将其加入问题治理流程。

## 1. 问题定义

现代实现已将走、跑、重力、跳跃、二段跳、落地和平台判定集中在 `HeroMovementSystem`，但“角色运行时如何创建移动模型、保留上一帧输入、调用移动、应用关卡环境、处理存活门禁并同步显示”仍由各场景或 bridge 分别编排。

问题本体不是底层物理公式已经重复，而是共享移动系统缺少统一的运行时接入边界。这使每个新关卡都可能重写一段玩家遍历和输入桥接，并可能遗漏跳跃、朝向、技能移动锁定、死亡门禁或上一帧输入。

如不治理，随着横向关、纵向关、移动平台、水下关和特殊重力关增加，重复接线和行为漂移将继续扩大。

## 2. 证据

- `src/systems/HeroMovementSystem.ts:50` 的 `createHeroMovement()` 和 `:72` 的 `updateHeroMovement()` 已是共享规则 owner，证明走/跑/跳/重力本身不必按关卡复制。
- `src/scenes/TestScene.ts:686` 的 `updateHeroMovement()` 自行遍历玩家、选择输入和上一帧输入、调用系统，再同步 sprite/label。
- `src/scenes/stage12/Stage12GameplayBridge.ts:143` 的 `updatePlayers()` 再次实现遍历、上一帧输入、存活门禁、调用系统和视图同步。
- Stage 1-2 初版只手工更新 x，没有调用共享移动系统，导致 P1 K 完全无法跳跃；`TASK-SLICE-128` 修正后才恢复。
- `src/systems/Stage12TraversalSystem.ts` 已将 Stage 1-2 的地面、镜头和停点右边界拆成关卡环境/遍历规则，但关卡 bridge 仍直接编排角色移动运行时。
- 现有至少两条正式集成路径直接调用 `updateHeroMovement()`，并且已出现一次用户可见的遗漏，因此不是未来假设或单文件小修。

## 3. 解决方案

建立一个关卡无关的“玩家队伍移动运行时” owner（最终命名在实施 task 中按统一语言确认）：

1. 共享运行时持有每个玩家的 `HeroMovementModel`、上一帧 `PlayerInputState` 和可移动/存活状态，统一调用 `HeroMovementSystem`。
2. 关卡只提供当帧环境快照：平台集合、移动边界、地面/掉落边界和必要的特殊环境标志。
3. 关卡流程只观察移动结果，据此处理镜头、停点、锁屏、出怪和通关；不再修改走/跑/跳的内部状态。
4. Phaser 显示对象同步保留在薄 view adapter，但 scene/bridge 不再自行保留上一帧输入或决定是否调用共享移动。
5. 先迁移 `TestScene` 与 Stage 1-2 两个已有消费者，以一横一纵两种场景验证接口；后续关卡只实现环境 provider 和关卡流程。
6. 如同类遗漏可能复发，在方案稳定后把“scene/bridge 不得直接拥有移动运行时”沉淀为 `code-quality-gates.md` 评审规则或自动搜索门禁。

治理边界：

- 不将镜头、停点、出怪或通关规则塞进通用移动系统。
- 不为此引入完整 ECS、事件总线或大范围场景重构。
- 不在本问题中校准所有关卡的地形数据；只建立统一接入边界并迁移现有消费者。

## 4. 测试方案

- 为共享玩家移动运行时增加纯系统测试：相同输入和环境快照必须得到相同位置、速度、朝向、跳跃次数和平台状态。
- 覆盖 1P/2P、按键边沿、持续移动、二段跳、落地、死亡/禁用玩家、动态边界变更和单向平台。
- 保留并运行 Stage 1-1/`TestScene` 移动回归、Stage 1-2 `stage12-traversal-tests.ts`、全量 `npm run test:systems` 和 `npm run build`。
- 用窄搜索确认迁移后只有共享 owner 直接调用 `updateHeroMovement()`；scene/bridge 不再各自保存 `previousInput` 或复制玩家移动遍历。
- 运行时手工验收至少覆盖：1-1 纵向场景、1-2 横向场景、1P/2P、K/小键盘 2 和每波锁屏边界。
- 方案落地后观察至少两个后续关卡接入样本，确认新关卡只提供环境/流程，没有复制移动运行时。

- `TASK-SLICE-128` 修正了 Stage 1-2 当前的 K、地面基线和停点触发，但只是修复一个消费者，不等于统一接入边界。
- `npm run check:workflow` 已通过：问题索引、游戏 task/feature-line 状态和 372 条资源标注保持一致；本次未修改 `src/`，因此没有将现有代码验证误记为治理方案已实施。

## 6. 关闭标准

PG-003 只有同时满足以下条件才可关闭：

- 已建立一个关卡无关的玩家队伍移动运行时 owner，统一持有移动模型、上一帧输入和可移动状态。
- `TestScene`/Stage 1-1 与 Stage 1-2 已迁移，scene/bridge 只提供关卡环境、流程和视图适配。
- 共享系统测试、现有关卡回归、全量系统测试和生产构建全部通过。
- 窄搜索或自动门禁能够证明 scene/bridge 没有新的直接移动运行时定义。
- 至少两个后续关卡接入样本未复制移动调度逻辑，且未再出现关卡遗漏跳跃/朝向/锁定的问题。
- 如风险会复发，`code-quality-gates.md` 或校验脚本已固化相应约束。

以下情况只能算部分缓解，不能关闭：

- 只保留现有 `HeroMovementSystem`，但每个关卡仍自行调用和保存上一帧输入。
- 只修复 Stage 1-2，没有迁移另一个消费者来验证抽象。
- 把镜头、停点、出怪或所有关卡差异塞入通用移动系统。
- 只通过当前测试，没有对后续关卡的生成约束或观察样本。
