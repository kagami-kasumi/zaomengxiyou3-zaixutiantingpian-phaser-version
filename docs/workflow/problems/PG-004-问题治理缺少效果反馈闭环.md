# PG-004 问题治理缺少效果反馈闭环

状态：规则已落地，运行观察中。

来源：2026-07-21 用户指出，已登记问题只能看到问题和方案，后续新代码没有统一回审旧方案的解决程度和防复发效果。

## 1. 问题定义

问题治理协议定义了问题、方案、测试和关闭标准，但没有统一规定后续任务如何判定适用旧问题、如何回写样本、如何处理复发，也没有自动校验所有 `PG-*` 都声明了反馈触发。

结果是问题可以长期停在“方案试行中”，却无法回答存量问题还剩多少、新代码是否再犯、没有样本是因为不适用还是问题定义过窄。

## 2. 证据

- PG-001 要求三个未来样本，但没有全局任务触发规则，关卡代码大量增长后只能人工推断“不适用”。
- PG-002 有强自动校验，但 Stage 1-2 仍在 K、地面基线和波次触发错误存在时关闭，证明“文档状态合法”不等于“关闭效果充分”。
- `validate-workflow.mjs` 原先只硬编码 PG-001/002；PG-003 文档存在且被索引后，校验仍不读取它。
- 旧模板只有 6 节，没有“适用触发与反馈记录”。

## 3. 解决方案

- 将问题文档扩展为 7 节，每个问题必须声明触发条件、效果检查和统一样本表。
- 代码/架构/工作流任务收尾前必须执行问题适用性扫描，对命中问题回写样本。
- 效果同时评估存量解决程度和增量防复发效果。
- 复发或方案不充分时强制退回治理；已关闭问题必须重开。
- 校验器动态发现全部 `PG-*.md`，校验索引、7 节结构、触发条件和反馈表，不再为每个 PG 手工硬编码。
- 立即回审 PG-001—03；对无关新代码明确记录不适用，对 Stage 1-2 关闭漏洞记录方案不充分。

## 4. 测试方案

- 运行 `npm run check:workflow`，确认全部 PG 动态发现并通过新结构校验。
- 使用临时负向样例验证：删除第 7 节、删除触发条件或反馈表时必须失败。
- 回审 PG-001—03 并检查其状态、样本和问题索引一致。
- 后续至少三个代码/架构任务都必须在交接中报告问题治理反馈，命中时回写对应 PG。

## 5. 测试结果

- `node --check tools/validate-workflow.mjs` 通过。
- `npm run check:workflow` 通过；校验器动态发现并验证 PG-001—04，而不是依赖固定文件清单。
- 校验器内置的三个负向样例均被拒绝：删除第 7 节、删除触发条件、删除反馈表任一情况都无法通过反馈契约检查。
- PG-001—03 已完成首次回审；其中 PG-002 的 Stage 1-2 样本结论为“方案不充分”，其状态已从运行观察退回“方案补强中”，证明反馈结论会改变治理状态而非只记日志。

当前协议、入口、动态校验和一次真实退回治理均已验证；尚缺三个后续变更的持续执行样本，因此保持运行观察中。

## 6. 关闭标准

- 所有 PG 都有可执行的适用触发、存量/增量效果检查和反馈表。
- 校验器动态发现 PG，结构负向样例可靠失败。
- 至少三个后续适用性扫描样本证明任务收尾会真正执行反馈闭环。
- 至少一次复发或模拟复发能正确促使问题退回治理/重开。
- 长期无适用样本的问题会被主动审计，不再无限期无证据观察。

仅补协议文案、仅增加一张空表，或仅运行一次当前校验，都只算部分缓解。

## 7. 适用触发与反馈记录

触发条件：

- 任何代码、架构、游戏 task 或工作流变更的收尾。
- 新增、修改、关闭或重开任何 `PG-*`。

效果检查：

- 检查任务交接是否报告 PG 适用性，命中问题是否回写样本。
- 检查结构校验是否动态覆盖全部问题文档，复发是否改变问题状态和后续方案。
