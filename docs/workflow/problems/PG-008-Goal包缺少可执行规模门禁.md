# PG-008 Goal 包缺少可执行规模门禁

状态：规则落地，效果观察中。

来源：用户复核 `GOAL-020` 后指出单次 Goal 仍然过长；执行记录显示该 Goal 虽满足“一 task、最多一次 compact”的旧规则，却实际同时处理完整关卡几何、波次、机关、Boss 真视觉、结果存档、资源标注和任务生成。

## 1. 问题定义

现有 Goal 规则只限制绑定 task 数和 compact 上限，没有限制单个 task 内的主工作包、资料族、owner 或独立验收批次数量。结果是 Goal 可以在形式上合规，但仍然占用过长执行时间和上下文，失去“有界交接包”的作用。

这是系统性问题：

- 任何复杂逆向或实现 task 都可以把多个独立子系统写进一个完成定义。
- compact 次数是工作已经膨胀后的滞后指标，不能用于生成阶段预防。
- “影响实现的未知为零”容易被误解为必须在单个逆向 Goal 内清零整条功能线的全部未知。
- 一个大 task 即使只绑定一个 Goal，也可能同时包含资源派生、领域实现、运行校准和关闭裁决。

## 2. 证据

- `GOAL-020 / TASK-SETTINGS-063` 同时闭合 Stage 2-2 场景/几何、25 刷怪点/54 怪、9 火焰、Monster16 36 个本体关键帧/六攻击对象 104 帧、结果/存档、14 条资源标注和下一任务生成。
- 该 Goal 使用一次 compact，形式上满足旧规则，但执行约 46 分钟，证明“最多一次 compact”不足以表示规模合理。
- 原 `GOAL-021 / TASK-SLICE-150` 又把 14 条资源、layout/traversal/fire/flow/Monster16 owners、五停点、54 怪、结果存档和完整 940×590 验收绑入一个 task，具备立即复发条件。
- `tools/validate-workflow.mjs` 只拒绝一个 Goal 绑定超过两个 task，不检查 task 的主工作包和预期 compact。

## 3. 解决方案

1. 新 Goal 默认必须预计 `0` 次上下文压缩；`goal-board.md` 只保存未完成 Goal，完成事实由绑定 task 的 `task-history.md` 记录。
2. 所有未完成 task 定义新增固定字段：
   - `规模预算：`
   - `主工作包：1` 或 `2`
   - `预计上下文压缩：0`
   - `独立验收批次：1` 或 `2`
   - `拆分触发：`
3. 可执行 task 最多两个可独立命名的主工作包、最多两个独立验收批次；超过即在执行前拆分，不以“同一关卡/同一页面”为合并理由。
4. `/goal` 开始后先做规模预检。若 task 实际范围超过声明，允许本次只完成任务重排和安全交接；不得为了“完成 Active Goal”继续实质实现。
5. 第一次 compact 成为拆分信号：只允许结束正在运行的检查、回写安全检查点和拆分剩余工作，不再读取新资料族、派生新资源或新增实现。
6. 逆向任务的“未知为零”只针对当前 Goal 的主工作包；整条功能线可以由多个连续 Goal 分别清零。
7. 将 Stage 2-2 当前实现拆为场景/机关、波次/普通怪、Monster16/结果、完整运行校准四个连续 Goal。

明确不做：

- 不按 token 数或分钟数机械关闭任务；这些只作为复核信号。
- 不把同一功能线拆成并行 Active Goal。
- 不回写或伪造历史 Goal 当时未声明的规模预算。

## 4. 测试方案

- `tools/validate-workflow.mjs` 拒绝：
  - 未完成 Goal 使用“最多 1 次”而不是“预计 0 次”；
  - `goal-board.md` 保留 `Done` Goal；
  - 未完成 task 缺少规模预算或拆分触发；
  - 主工作包超过 2；
  - 预计 compact 非 0；
  - 独立验收批次超过 2。
- 内置正样例验证一个主工作包、0 compact、一个验收批次可通过。
- 内置负向样例逐项删除或超限，必须失败。
- `npm run check:workflow` 验证 Stage 2-2 拆分后仍保持唯一 Active line、唯一 Active Goal、唯一 Ready/推荐 task。

## 5. 测试结果

- `GOAL-021 / TASK-SLICE-150` 已在开始实现前拆为 `TASK-SLICE-150A..150D` 和 `GOAL-021..024`。
- 每个新 task 只保留一至两个主工作包、0 compact 预期和一至两个验收批次。
- `node --check tools/validate-workflow.mjs` 通过。
- `npm run check:workflow` 通过：5 个未完成 task / 5 个定义、唯一推荐 `TASK-SLICE-150A`、唯一 Active `GOAL-021`；规模预算正样例通过，缺字段、工作包超限、compact 非 0、验收批次超限四类负向样例均被拒绝。
- 仅保留既有 `PlayerSlot` 禁止别名 warning。

## 6. 关闭标准

PG-008 只有同时满足以下条件才可关闭：

- 规模字段和新 Goal 的 0 compact 预算由自动校验强制执行。
- 至少三个连续新 Goal 在不 compact 的情况下完成，且没有在执行中再次拆分。
- 至少覆盖一个逆向 Goal、一个实现 Goal和一个运行校准 Goal。
- 后续用户复核没有再次指出 Goal 过长或交付边界混杂。

只增加文案、只拆当前 Goal、或只让校验器检查字段存在，都只算部分缓解。

## 7. 适用触发与反馈记录

触发条件：

- 新建、拆分或激活任何 Goal/task。
- `/goal` 执行发生 compact。
- 一个 task 同时要求新资料族、资源派生、多个 owner 和端到端运行验收。
- 用户认为 Goal 过长、上下文消耗异常或交付边界混杂。

效果检查：

- 对照 task 声明的主工作包、compact 和验收批次检查实际执行。
- 检查是否在 compact 后继续扩张范围。
- 检查拆分后是否保持同线串行、唯一 Active Goal 和可独立交接产物。
