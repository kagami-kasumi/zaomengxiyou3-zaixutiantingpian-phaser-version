# PG-007 UI 原生化缺少统一门禁

状态：规则已落地，运行观察中。

来源：2026-07-23 用户在炼丹炉原生按钮整改后进一步指出：技能系统及后续系统不得把原版页面当背景后再覆盖一套现代 UI；已实现页面需要迁移，未实现页面需要提前规避。

## 1. 问题定义

项目已经能够从恢复 SWF 中取得原版页面、中文字、按钮、按钮帧和局部子页，但“接入真资源”尚未等价为“UI 原生化”。多个正式功能页仍把真资源作为背景，再使用 Phaser 矩形、现代标题、通用按钮和暗色面板重建一层视觉与交互。

本项目中的 **UI 原生化** 定义为：

- 原始 SWF 中的页面图片、中文字、按钮显示对象、按钮帧、原始布局和已有文本/图标槽位是视觉权威。
- 原图片中的中文字直接承担按钮视觉；允许叠加透明命中区处理事件，但不得重新绘制替代文字、边框、底色或选中标签。
- 原版按钮存在 up/over/down/selected 等状态时，现代实现复用对应原生状态，不用通用矩形按钮模拟。
- 动态数值、物品图标和状态进入原版预留槽位；不能因为现代数据更方便展示而新增覆盖原图的外层 UI。
- 原版确实没有对应表现时，现代补充必须单独标记为“现代设计选择”，记录证据缺口、最小视觉边界并经过用户验收。

该问题同时影响两类工作：

1. **存量迁移**：技能、宠物、法宝等已经形成现代覆盖层的页面，需要按系统逐个逆向和整改。
2. **增量防复发**：后续 UI task 若没有前置门禁，会继续把“加载了真背景”误写成“真 UI 已完成”。

不治理的后果是：每个系统先造一套临时现代 UI，用户验收后再返工；原版坐标、按钮状态和交互语义被覆盖；任务与功能线可能在视觉合同未满足时提前关闭。

## 2. 证据

- `src/scenes/feature-ui/FormalSkillPageView.ts` 当前包含 3 个 `scene.add.rectangle`、5 个 `scene.add.text` 和 9 组通用 `button(scene, ...)` 调用；其中包括 940×590 暗层、900×548 外层面板和现代标题。
- `src/scenes/feature-ui/FormalPetPageView.ts` 当前包含 3 个矩形、7 处现代文字和 11 组通用按钮。
- `src/scenes/feature-ui/FormalMagicWeaponPageView.ts` 当前包含 6 个矩形、11 处现代文字和 6 组通用按钮。
- `src/scenes/feature-ui/FormalWorkshopPageView.ts` 经 `TASK-SLICE-142` 后虽为 0 个矩形和 0 组通用按钮，但仍以扁平 119 容器叠加统一中心子页、现代 P1/P2 文字、线性背包文本和现代状态文案；原 AS3 实际动态创建 `BackPackElement`、25 个 `PackThings` 和角色选择 MovieClip。该反例证明静态计数、透明命中和“真背景已加载”仍不足以证明 UI 原生化。
- `StrengthEquipment.as` 已给出四子页、背包和角色入口的运行时 child/坐标，`BackPackElement.as` 已给出 5×5 动态格子，但此前实现/验收没有把它们组织成完整显示列表清单，也没有用原版运行画面做同尺寸叠图或像素差异。
- `docs/reverse-engineering/full-function-ui-index.md` 已把技能总页及三个子页记录为真资源并标记正式实现完成，但现有 view 仍用外层现代面板、标题和通用按钮承载主要交互，说明“真资源接入”和“UI 原生化完成”之间缺少独立验收门禁。
- 用户明确要求将该问题统一命名为“UI 原生化”，并要求同时解决已造 UI 的迁移与后续系统的预防。

以上计数只用于定位高风险页面，不把 `rectangle` 或 `text` 的存在自动判为违规。最终判断必须结合恢复 SWF、原版槽位、按钮时间轴和 940×590 运行画面。

## 3. 解决方案

### 3.1 建立 UI 原生化合同

在任务生成、逆向、实现和评审入口统一要求 UI task 记录：

- 权威源 SWF、SymbolClass、MovieClip、character id 和目标帧。
- 根页面与子页的注册点、嵌套矩阵、可见边界、940×590 坐标映射。
- 原生中文字、按钮实例、up/over/down/selected 状态和命中区。
- 动态文本、图标、列表、分页、确认/返回等原版槽位。
- 允许新增的现代视觉元素清单；未列出的现代标题、面板、标签、按钮和暗罩默认禁止。
- 原生化的自动验证与逐状态浏览器验收计划。
- 完整显示列表清单：root/child/depth/父子关系、注册点、嵌套矩阵、mask/filter/blend、TextField、按钮状态、动态 `addChild` 和命中区。
- 原版视觉基准：可追溯 Flash 截图/录屏或原 SWF 渲染的版本、入口、状态、舞台尺寸与裁切；现代截图不得反向充当原版基准。
- 同尺寸并排/半透明叠图、稳定区域像素或边缘差异、逐对象差异清单和字体栅格化容差说明。

### 3.2 存量页面迁移

先建立正式功能页存量矩阵，对每页分类：

- 已原生化。
- 真资源已接入但仍有替代覆盖层。
- 仅行为原型。
- 原版资源或交互证据未闭合。

技能系统作为炼丹炉后的第一个整改样本，仍拆为独立证据 task 和实现 task。宠物、法宝、背包等按各自系统生成后续小任务，不一次性重构全部 UI，不在 PG 文档中直接修改游戏代码。

### 3.3 增量防复发

- `task-generation.md`：UI task 必须包含“UI 原生化合同”和允许的现代例外。
- `reverse-engineering-protocol.md`：涉及 UI 时，六段证据链必须覆盖文字、按钮状态、命中区和动态槽位，而不只记录整页图片尺寸。
- `code-quality-gates.md` / `review-protocol.md`：把“真资源作为背景并覆盖替代 UI”列为阻塞验收的问题。
- `validate-workflow.mjs` 只校验任务/文档是否包含必填合同，不用静态关键词冒充视觉正确性。
- `validate-workflow.mjs` 必须用正/负样例拒绝缺少显示列表、原版视觉基准、现代例外或逐状态验收的 UI task；视觉本身仍由证据和人工/图像对照判断。
- 运行验收必须从正式入口逐页检查原生按钮的 up/over/down/selected、命中边界、动态槽位和返回流程。

### 3.4 Skill 边界

当前不以 skill 代替治理门禁。Skill 不能强制项目任务、评审和校验器遵守合同，也容易在项目规则变化后漂移。待炼丹炉、技能页及至少一个后续页面形成稳定样本后，可另行评估是否将成熟方法抽成跨项目的“Flash UI 原生化”skill。

### 3.5 调度边界

登记 PG-007 不改变游戏任务 `WIP=1`。当前 `LINE-STAGE-2-1` / `GOAL-009` 继续保持 Active；技能页整改只有在用户明确批准暂停或重排当前功能线后，才进入游戏 task/Goal 系统。

## 4. 测试方案

- 存量审计：逐个统计 `Formal*PageView.ts` 中现代矩形、现代标题、通用按钮和 extracted asset 的组合，并人工判定是否属于替代覆盖层。
- 证据门禁：检查 UI 逆向文档是否包含原生文字、按钮状态、命中区、注册点、矩阵、动态槽位、证据分级和反证条件。
- 显示列表门禁：检查 root/child/depth、嵌套变换、mask/filter、TextField、动态 child 与现代资源一一映射。
- 视觉基准门禁：检查原版基准的来源/状态/尺寸，以及同尺寸叠图、像素/边缘差异、容差和逐对象差异清单。
- 自动合同：验证原生资源 key、按钮状态资源、940×590 坐标、相邻命中区不重叠和透明命中区不绘制替代视觉。
- 回归测试：业务事务、P1/P2 owner、切页返还、关闭、保存和重载不得因视图原生化而变化。
- 运行观察：从正式入口逐页保存 940×590 的 normal/hover/pressed/selected 和关键业务状态截图，检查 console 无 error/warning。
- 工作流校验：运行 `npm run check:workflow` 和 `git diff --check`；实际代码整改另运行 `npm run check:structure`、相关专项、`npm run test:systems` 与 `npm run build`。

## 5. 测试结果

当前已完成规则落地，存量迁移仍待执行：

- 炼丹炉从“参考样本”退回存量反例：功能/路由已验证，但显示列表、右侧 5×5 背包、角色入口和原版视觉基准没有闭合，不能再标记为视觉原生化完成。
- 技能、宠物和法宝页均命中“真资源背景 + 现代覆盖层”的高风险模式，技能页被用户明确指定为下一整改对象。
- `reverse-engineering-protocol.md`、`task-generation.md`、`code-quality-gates.md`、`review-protocol.md`、Agent 入口和 `validate-workflow.mjs` 已接入显示列表、原版基准、现代例外和逐状态差异门禁。
- 技能两个 Planned task 已补齐 UI 原生化合同；尚未建立全量页面矩阵，尚无技能页或第二个存量页面整改结果。
- `npm run check:workflow` 已通过；UI 合同六字段正样例通过，逐字段删除的六个负向样例均被拒绝，逆向协议新增 UI 门禁/原版基准退化样例也被拒绝。

因此 PG-007 进入“规则已落地，运行观察中”，不能以协议/校验器完成或任何单个页面样本关闭。

## 6. 关闭标准

- “UI 原生化”定义进入任务生成、逆向、实现和评审门禁，且工作流校验能拒绝缺少原生化合同的新增 UI task。
- 逆向门禁明确覆盖完整显示列表、原版视觉基准、逐状态叠图/像素或边缘差异和可见对象差异清单。
- 正式功能页存量矩阵覆盖所有已实现和计划中的 UI 页面，并为每个未原生化页面留下游戏 task/排期或明确排除理由。
- 技能页完成原生文字/按钮/布局迁移，业务、双 owner、存档和正式路径回归通过。
- 至少再完成一个存量页面整改样本，证明方案可复用且没有演变为一次性炼丹炉特例。
- 至少三个后续 UI task 在增量门禁下没有新增替代外层 UI；命中现代例外时均有证据、最小边界和用户验收。
- 自动检查、系统测试、构建、940×590 逐状态视觉验收和 console 检查全部通过。
- Skill 是否创建不是 PG-007 的关闭条件；只有治理方法稳定且确认存在跨项目复用需求时才单独决定。

以下只能算部分缓解，不能关闭：

- 只删除暗色背景，但仍用现代文字/通用按钮覆盖原生中文字。
- 只把按钮改成透明命中区，却没有复用原版按钮状态或确认命中几何。
- 只有整页背景、静态源码计数、路由/事务测试或现代页面截图，没有完整显示列表和原版视觉基准。
- 只整改技能页，没有建立后续任务门禁。
- 只增加文档规范，没有存量迁移样本和增量防复发样本。

## 7. 适用触发与反馈记录

触发条件：

- 新增或修改 `src/scenes/feature-ui/*View.ts`、正式 HUD/菜单/功能页视图或 UI host。
- 新增、拆分、实现、评审或关闭任何带真 UI、原版页面、按钮、菜单、HUD、overlay 的 task/Goal/功能线。
- 接入恢复 SWF 页面、按钮、文字、时间轴帧或把 extracted asset 用作整页背景。
- diff 同时出现 extracted UI asset 与新的全屏矩形、现代标题、通用按钮或替代标签。
- 用户反馈页面“不是原生 UI”、按钮未直接使用原图、布局被现代外层 UI 覆盖。

效果检查：

- 窄查目标 view 的矩形、文字、通用按钮、透明命中区和原生资源组合，人工区分动态槽位与替代覆盖层。
- 检查对应 task/逆向文档是否包含 UI 原生化合同、允许的现代例外和逐状态运行验收。
- 检查存量矩阵剩余缺口、已整改页面数量以及最近三个适用 UI task 是否复发。
