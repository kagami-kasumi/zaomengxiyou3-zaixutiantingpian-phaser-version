# 问题治理协议

本文规定系统性工程问题如何被定义、确认、治理和关闭。它属于 AI 协作脚手架，不记录具体游戏复现任务。

## 适用范围

使用本协议的场景：

- 评审发现跨文件重复、模块耦合、边界漂移、测试盲区、数据源分裂或生命周期规则不统一。
- 用户或 reviewer 明确提出“这是一个问题”，需要人和 AI 共同确认后再治理。
- 一个局部 bug 暴露出更深层的系统性缺口，例如桥接层承载领域规则、共享数据表复制到多个系统。
- 代码质量门禁或评审协议发现的风险需要被拆成可验证的治理对象。

不使用本协议的场景：

- typo、注释、单个常量、单文件局部 bug 或无需协商的小修。
- 已有明确游戏 task 的正常实现流程。
- 单纯玩法逆向事实摘录，除非该事实暴露出现代代码的系统性问题。

## 问题定义

本项目中的“问题”不是任意代码瑕疵，而是会持续伤害系统演进能力的结构性缺口。一个问题通常具备以下至少两项特征：

- 影响多个文件、多个角色、多个系统或多条调用路径。
- 修改一个业务事实需要同步改多处代码。
- 不同模块对同一规则的表达不一致。
- 当前实现能跑，但继续扩展会显著增加维护成本。
- 测试只能证明局部行为，不能覆盖真实集成路径。
- 问题背后存在职责边界、数据归属、生命周期或验证策略不清。

示例：

- MP 表、伤害公式或输入 helper 在多个技能系统中重复定义。
- `clampLevel` / `findJustPressedSkillSlot` 每个文件独立实现且签名不一致。
- `HeroSkillModel` 同时承载全部角色 runtime，导致初始化、reset 和类型边界耦合。
- bridge 层消费领域规则，系统函数无法独立使用。

非示例：

- 某一行拼写错误。
- 单个资产 key 写错。
- 一个局部测试断言缺少边界值。

这些局部事项可以直接修；只有当它们暴露出系统性缺口时，才升级为问题治理。

## 治理记录结构

每个问题治理记录应包含 6 部分：

1. **问题定义**
   - 问题是什么。
   - 为什么它是系统性问题，而不是局部修复项。
   - 不治理会造成什么后果。

2. **证据**
   - 文件、行号、重复次数、结构 warning、测试缺口或调用链。
   - 证据必须可复查，避免只写“代码复杂”“耦合严重”。

3. **解决方案**
   - 准备怎么改。
   - 改到什么边界为止。
   - 明确不做什么，避免顺手扩大成无边界重构。

4. **测试方案**
   - 用什么证明方案正确。
   - 可包含系统测试、构建、结构检查、搜索验证、人工试玩点或评审复查。
   - 对去重类问题，测试方案应包含搜索验证，不只依赖 `npm run test:systems`。

5. **测试结果**
   - 实际运行了什么。
   - 是否通过。
   - 剩余 warning、风险或未覆盖点是什么。

6. **关闭标准**
   - 什么条件下可标记为已治理。
   - 什么条件下只能算部分缓解。
   - 是否需要后续 task、评审或校验脚本规则。

## 治理流程

1. **提出问题**：用户、reviewer 或实现 agent 可以提出候选问题，但必须先写清问题定义和证据。
2. **共同确认**：涉及架构边界、系统拆分或较大范围改动时，先让用户确认问题是否值得治理。
3. **选择落点**：
   - 游戏复现工作进入 `docs/tasks/task-board.md`。
   - 代码质量、协作协议、校验脚本或评审规则进入 `docs/workflow/`。
   - 轻量修复可以直接在当前改动中完成，但仍要保留证据和测试结果。
4. **执行治理**：保持范围收敛，一次只处理一个清晰问题或一个可验收子问题。
5. **验证关闭**：按测试方案运行命令和搜索；证据不足时不能宣称关闭。
6. **沉淀规则**：如果同类问题可能复发，优先把关闭标准转为质量门禁、评审规则或校验脚本检查。

## 问题记录

### PG-001 共享技能规则重复定义

状态：方案试行中。

来源：`docs/评审/five-roles-review-2025-06-26.md` 的 CH1-CH3，以及用户确认“重复定义”才是问题；“数值校准需要跨文件同步、容易遗漏”只是后果。

#### 1. 问题定义

五角色技能系统中存在共享规则重复定义：

- 共享 MP 表、伤害公式和段数表在多个技能系统中重复定义。
- `clampLevel` / `findJustPressedSkillSlot` 等基础 helper 在多个文件中独立实现，且签名和行为细节不一致。

问题不是“调数值麻烦”，而是同一条共享规则存在多个现代定义源。数值校准需要跨文件同步、容易遗漏，是重复定义带来的影响，不是问题本体。

`HeroSkillModel` runtime 全量承载属于另一个耦合问题，不并入 PG-001；它可作为后续独立问题治理记录。

#### 2. 证据

- 评审记录指出 `consumeMpByLevel` 在 15+ 个文件中逐字重复。
- `skillFactorBase` / `skillFactorPerLevel` / `fixedDamageCount` 在多个角色技能系统中重复定义。
- `clampLevel`、`clampLevelOrZero`、`findJustPressedSkillSlot` 存在多个局部实现。

#### 3. 解决方案

解决方案不是单次抽取代码，而是一条约束 AI 生成代码的 harness 规则：**共享定义所有权约束**。

AI 在新增或修改代码时，如果准备定义满足以下条件之一的内容，必须先定位现有 owner，而不是直接复制到当前文件：

- 跨角色、跨技能、跨系统通用的数据表，例如 MP 表、通用伤害段数表、通用技能系数。
- 跨文件可复用的纯函数 helper，例如等级裁剪、技能槽输入检测、距离计算。
- 同一业务规则的第二份现代定义。

执行规则：

1. 先用 `rg` 搜索稳定短关键词，确认是否已有 owner。
2. 如果已有 owner，必须复用或扩展 owner，不得在新文件重复定义。
3. 如果没有 owner，但预计会有两个及以上消费者，应先创建 owner 模块，再让消费者引用。
4. 如果只服务单个技能或单个角色，可保留局部定义，但注释或命名应体现其局部性。
5. 评审时发现新增重复定义，应退回修改；不能只靠“以后再抽”通过。

当前候选 owner：

- `SkillTuning.ts`：共享 MP 表、通用伤害段数表、通用技能系数。
- `SkillInputUtils.ts`：技能槽输入检测。
- `SkillMathUtils.ts`：等级裁剪、通用数学 helper。

这是一条生成约束，不是一次性重构任务。当前代码抽取只能作为首个样本，不能单独证明 harness 方案可行。

#### 4. 测试方案

本问题测试的是 harness 约束是否能影响未来 AI 生成代码，而不是只测试当前代码是否能跑。

短期检查：

- 使用 `rg` 搜索旧重复模式，例如局部 `consumeMpByLevel`、局部 `findJustPressedSkillSlot`、重复 `skillFactorBase` / `skillFactorPerLevel`。
- 运行 `npm run check:workflow`，确认问题治理规则仍被脚手架接住。

长期可行性验证：

- 后续至少 3 次涉及技能、数值、helper 或跨角色共享规则的代码生成任务，都必须在收尾时检查是否遵守“共享定义所有权约束”。
- 每次样本记录任务、日期、涉及文件、是否新增重复定义、验证命令或搜索关键词。
- 3 次样本全部没有新增重复定义，才说明该 harness 方案初步可行。
- 任意一次样本新增重复定义，则判定当前方案不可行或不充分，需要升级方案，例如加入更强的校验脚本、模板约束或评审必检项。

#### 5. 测试结果

当前结果：

- 已完成一次历史样本：五角色技能评审整改中抽取了 `SkillTuning.ts`、`SkillInputUtils.ts` 和 `SkillMathUtils.ts`，并通过 `npm run check:structure`、`npm run test:systems` 和 `npm run build`。
- 该样本只能证明“旧重复定义可以被修复”，不能证明“harness 能约束未来生成代码不再重复定义”。

#### 6. 关闭标准

满足以下条件时，PG-001 才可视为关闭：

- 问题定义稳定为“重复定义”，不把影响误写成问题本体。
- `code-quality-gates.md` 或同级脚手架文档明确声明共享定义所有权约束。
- 后续至少 3 次相关代码生成任务均未新增共享规则重复定义。
- 每次样本都有搜索或评审证据，而不是只看系统测试通过。
- 若 3 次样本中出现新增重复定义，必须重新设计方案，不能关闭本问题。

## 与其他协议的关系

- `review-protocol.md` 负责发现和排序问题；本协议负责把系统性发现变成可治理、可关闭的对象。
- `code-quality-gates.md` 负责硬性质量门禁；本协议可以把反复出现的问题沉淀成新的门禁。
- `task-generation.md` 负责生成游戏任务；只有明确属于游戏复现或现代实现缺口的问题才进入任务看板。
- `governance-log.md` 记录脚手架规则变化；问题治理协议本身的变更必须写入治理日志。

```md

### 1. 问题定义
共享 MP 表、伤害段数表、技能系数在多个角色系统中重复定义。重复定义是问题本体；数值校准需要跨文件同步、容易遗漏是后果。

### 2. 证据
- `consumeMpByLevel` 在 15+ 个文件中重复。
- `skillFactorBase` / `skillFactorPerLevel` 多处逐字重复。

### 3. 解决方案
建立共享定义所有权约束：AI 生成代码前先搜索 owner；已有 owner 必须复用；预计有两个及以上消费者时先创建 owner；局部定义必须保持局部边界。

### 4. 测试方案
- 运行 `npm run test:systems`。
- 运行 `npm run build`。
- 用 `rg` 搜索旧重复常量。

### 5. 测试结果
记录后续至少 3 次相关代码生成任务是否新增重复定义；3 次全部正常才说明 harness 方案初步可行。

### 6. 关闭标准
共享定义所有权约束已写入脚手架；后续至少 3 次相关代码生成任务未新增重复定义；每次都有搜索或评审证据。
```
