# 问题治理协议

本文规定系统性工程问题如何被定义、确认、治理和关闭。它属于 AI 协作脚手架，不记录具体游戏复现任务。

## 适用范围

使用本协议的场景：

- 评审发现跨文件重复、模块耦合、边界漂移、测试盲区、数据源分裂或生命周期规则不统一。
- 用户或 reviewer 明确提出“这是一个问题”，需要人和 AI 共同确认后再治理。
- 一个局部 bug 暴露出更深层的系统性缺口，例如桥接层承载领域规则、共享数据表复制到多个系统。
- 代码质量门禁或评审协议发现的风险需要被拆成可验证的治理对象。

不使用本协议的场景：

- typo、注释、单个常量、单文件局部 bug 或无需协商的小修。
- 已有明确游戏 task 的正常实现流程。
- 单纯玩法逆向事实摘录，除非该事实暴露出现代代码的系统性问题。

## 问题定义

本项目中的“问题”不是任意代码瑕疵，而是会持续伤害系统演进能力的结构性缺口。一个问题通常具备以下至少两项特征：

- 影响多个文件、多个角色、多个系统或多条调用路径。
- 修改一个业务事实需要同步改多处代码。
- 不同模块对同一规则的表达不一致。
- 当前实现能跑，但继续扩展会显著增加维护成本。
- 测试只能证明局部行为，不能覆盖真实集成路径。
- 问题背后存在职责边界、数据归属、生命周期或验证策略不清。

示例：

- MP 表、伤害公式或输入 helper 在多个技能系统中逐字重复。
- `clampLevel` / `findJustPressedSkillSlot` 每个文件独立实现且签名不一致。
- `HeroSkillModel` 同时承载全部角色 runtime，导致初始化、reset 和类型边界耦合。
- bridge 层消费领域规则，系统函数无法独立使用。

非示例：

- 某一行拼写错误。
- 单个资产 key 写错。
- 一个局部测试断言缺少边界值。

这些局部事项可以直接修；只有当它们暴露出系统性缺口时，才升级为问题治理。

## 治理记录结构

每个问题治理记录应包含 6 部分：

1. **问题定义**
   - 问题是什么。
   - 为什么它是系统性问题，而不是局部修复项。
   - 不治理会造成什么后果。

2. **证据**
   - 文件、行号、重复次数、结构 warning、测试缺口或调用链。
   - 证据必须可复查，避免只写“代码复杂”“耦合严重”。

3. **解决方案**
   - 准备怎么改。
   - 改到什么边界为止。
   - 明确不做什么，避免顺手扩大成无边界重构。

4. **测试方案**
   - 用什么证明方案正确。
   - 可包含系统测试、构建、结构检查、搜索验证、人工试玩点或评审复查。
   - 对去重类问题，测试方案应包含搜索验证，不只依赖 `npm run test:systems`。

5. **测试结果**
   - 实际运行了什么。
   - 是否通过。
   - 剩余 warning、风险或未覆盖点是什么。

6. **关闭标准**
   - 什么条件下可标记为已治理。
   - 什么条件下只能算部分缓解。
   - 是否需要后续 task、评审或校验脚本规则。

## 治理流程

1. **提出问题**：用户、reviewer 或实现 agent 可以提出候选问题，但必须先写清问题定义和证据。
2. **共同确认**：涉及架构边界、系统拆分或较大范围改动时，先让用户确认问题是否值得治理。
3. **选择落点**：
   - 游戏复现工作进入 `docs/tasks/task-board.md`。
   - 代码质量、协作协议、校验脚本或评审规则进入 `docs/workflow/`。
   - 轻量修复可以直接在当前改动中完成，但仍要保留证据和测试结果。
4. **执行治理**：保持范围收敛，一次只处理一个清晰问题或一个可验收子问题。
5. **验证关闭**：按测试方案运行命令和搜索；证据不足时不能宣称关闭。
6. **沉淀规则**：如果同类问题可能复发，优先把关闭标准转为质量门禁、评审规则或校验脚本检查。

## 问题记录

### PG-001 技能系统共享规则重复与 HeroSkillModel runtime 耦合

状态：已治理，待提交。

来源：`docs/评审/five-roles-review-2025-06-26.md` 的 CH1-CH5，以及用户确认“这是系统性问题，不是局部实现问题”。

#### 1. 问题定义

五角色技能系统中存在三类系统性问题：

- 共享 MP 表、伤害公式和段数表在多个技能系统中重复定义。
- `clampLevel` / `findJustPressedSkillSlot` 等基础 helper 在多个文件中独立实现，且签名和行为细节不一致。
- `HeroSkillModel` 同时承载全部 5 个角色的多个 runtime 字段，创建和 reset 时全量初始化，导致类型、生命周期和角色边界耦合。

这些不是单个技能 bug，而是数据源、输入规则、等级裁剪和 runtime 生命周期的归属不清。后续继续扩展技能或校准数值时，任何一个共享规则变化都可能需要跨文件同步修改，遗漏风险高。

#### 2. 证据

- 评审记录指出 `consumeMpByLevel` 在 15+ 个文件中逐字重复。
- `skillFactorBase` / `skillFactorPerLevel` / `fixedDamageCount` 在多个角色技能系统中重复定义。
- `clampLevel`、`clampLevelOrZero`、`findJustPressedSkillSlot` 存在多个局部实现。
- `HeroSkillSystem.ts` 曾达到结构 warning 阈值附近，并承载五角色 runtime 初始化和 reset 逻辑。
- 默认测试配装只覆盖每个角色部分技能，集成场景无法直接触达完整技能树。

#### 3. 解决方案

- 新增 `SkillTuning.ts`，作为共享 MP 表、共享伤害段数表和通用技能系数的单一来源。
- 新增 `SkillInputUtils.ts` 和 `SkillMathUtils.ts`，统一技能槽输入检测、等级裁剪和距离计算 helper。
- 新增 `HeroSkillRuntimeAccessors.ts`，将角色 runtime 改为按 heroId 懒访问，`HeroSkillModel` reset 时只清理 runtime cache。
- 为五角色测试场景补充多套默认技能配装，使两个技能树行都能在集成场景中触达。
- 保留角色特有固定伤害表和调优表，不借问题治理名义做无边界全量重构。

#### 4. 测试方案

- 运行 `npm run check:structure`，确认目标文件不再触发本问题相关结构 warning。
- 运行 `npm run test:systems`，确认技能系统行为测试通过。
- 运行 `npm run build`，确认 TypeScript 和生产构建通过。
- 使用 `rg` 搜索旧重复模式，例如局部 `consumeMpByLevel`、局部 `findJustPressedSkillSlot`、硬编码 `* 1.27`、旧 `_tmp` 资产名和零尺寸事件投射物。
- 新增/扩展系统测试覆盖 Hero runtime 懒创建、默认配装完整性、Role1 shadow、Role2 passive、Role3 防御、Role5 jrjl-sword 交互。

#### 5. 测试结果

- `npm run check:structure` 通过；剩余 warning 为既有非本问题目标文件。
- `npm run test:systems` 通过。
- `npm run build` 通过；仅 Vite 大 chunk 提醒。
- 搜索确认旧局部 helper 和关键重复模式不再残留在目标范围。
- `HeroSkillSystem.ts`、`Role1BasicSkillSystem.ts` 和 `Role5SkillSystem.ts` 均降到结构 warning 阈值以下。

#### 6. 关闭标准

满足以下条件时，PG-001 可视为关闭：

- 共享 MP 表、通用伤害段数表和通用技能系数只有一个现代数据源。
- 技能输入槽检测和等级裁剪通过共享 helper 表达。
- `HeroSkillModel` 不再在创建和 reset 时全量实例化所有角色 runtime。
- 默认测试配装能覆盖五角色两行技能树。
- 结构检查、系统测试和构建通过。
- 当前治理改动完成代码 review 并提交。

## 与其他协议的关系

- `review-protocol.md` 负责发现和排序问题；本协议负责把系统性发现变成可治理、可关闭的对象。
- `code-quality-gates.md` 负责硬性质量门禁；本协议可以把反复出现的问题沉淀成新的门禁。
- `task-generation.md` 负责生成游戏任务；只有明确属于游戏复现或现代实现缺口的问题才进入任务看板。
- `governance-log.md` 记录脚手架规则变化；问题治理协议本身的变更必须写入治理日志。

## 模板

```md
## 问题：技能系统共享数值重复

### 1. 问题定义
MP 表、伤害段数表、技能系数在多个角色系统中重复定义，导致数值校准需要跨文件同步，容易遗漏。

### 2. 证据
- `consumeMpByLevel` 在 15+ 个文件中重复。
- `skillFactorBase` / `skillFactorPerLevel` 多处逐字重复。
- Role5 另有独立副本。

### 3. 解决方案
抽取 `SkillTuning.ts`，保留角色局部固定伤害表，只把真正共享的表和公式合并为单一来源。

### 4. 测试方案
- 运行 `npm run test:systems`。
- 运行 `npm run build`。
- 用 `rg` 搜索旧重复常量。

### 5. 测试结果
- `npm run test:systems` 通过。
- `npm run build` 通过。
- 搜索确认共享 MP/公式只剩单一数据源。

### 6. 关闭标准
共享 MP/公式无重复定义；角色特有表允许保留；后续新增技能必须复用共享模块。
```
