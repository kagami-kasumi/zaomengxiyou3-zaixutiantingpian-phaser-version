# 问题治理协议

本文规定系统性工程问题如何被定义、确认、治理和关闭。它属于 AI 协作脚手架，不记录具体游戏复现任务。每个已登记问题必须拆到 `docs/workflow/problems/` 下的独立文档；本文件只维护通用协议、问题索引和模板。

## 适用范围

使用本协议的场景：

- 评审发现跨文件重复、模块耦合、边界漂移、测试盲区、数据源分裂或生命周期规则不统一。
- 用户或 reviewer 明确提出“这是一个问题”，需要人和 AI 共同确认后再治理。
- 一个局部 bug 暴露出更深层的系统性缺口，例如桥接层承载领域规则、共享数据表复制到多个系统。
- 代码质量门禁或评审协议发现的风险需要被拆成可验证的治理对象。

不使用本协议的场景：

- typo、注释、单个常量、单文件局部 bug 或无需协商的小修。
- 已有明确游戏 task 的正常实现流程。
- 单纯玩法逆向事实摘录，除非该事实暴露现代代码的系统性问题。

## 问题定义

本项目中的“问题”不是任意代码瑕疵，而是会持续伤害系统演进能力的结构性缺口。一个问题通常具备以下至少两项特征：

- 影响多个文件、多个角色、多个系统或多条调用路径。
- 修改一个业务事实需要同步改多处代码。
- 不同模块对同一规则的表达不一致。
- 当前实现能跑，但继续扩展会显著增加维护成本。
- 测试只能证明局部行为，不能覆盖真实集成路径。
- 问题背后存在职责边界、数据归属、生命周期或验证策略不清。

局部拼写、单个资产 key 或一个边界断言缺失可以直接修；只有暴露出系统性缺口时才升级为问题治理。

## 治理记录结构

每个问题治理文档必须包含 6 部分：

1. **问题定义**：问题本体、系统性影响和不治理的后果。
2. **证据**：文件、行号、重复次数、结构 warning、测试缺口或调用链。
3. **解决方案**：准备怎么改、改到什么边界、明确不做什么。
4. **测试方案**：证明方案正确所需的系统测试、构建、搜索、人工验收或评审复查。
5. **测试结果**：实际运行结果、剩余 warning、风险和未覆盖点。
6. **关闭标准**：完整关闭、部分缓解和需要沉淀为门禁的条件。

问题文档还必须在开头写明稳定 `PG-*` id、状态和来源。一个问题只占一个文档；不在总协议中继续堆叠问题正文。

## 治理流程

1. **提出问题**：先写清问题定义和证据。
2. **共同确认**：涉及架构边界、系统拆分或较大范围改动时，由用户确认是否值得治理。
3. **建立文档**：在 `docs/workflow/problems/` 新建一个独立 `PG-*` 文档，并加入下方索引。
4. **选择落点**：游戏复现工作进入 `docs/tasks/`；协作协议、校验脚本和评审规则进入 `docs/workflow/`。
5. **执行治理**：保持范围收敛，一次只处理一个清晰问题或一个可验收子问题。
6. **验证关闭**：按测试方案运行检查；证据不足时不能宣称关闭。
7. **沉淀规则**：同类问题可能复发时，把关闭标准转为质量门禁、评审规则或自动校验。

## 与其他协议的关系

- `review-protocol.md` 负责发现和排序问题；本协议负责把系统性发现变成可治理、可关闭的对象。
- `code-quality-gates.md` 负责硬性质量门禁；问题治理可以把反复出现的问题沉淀成新门禁。
- `task-generation.md` 负责生成游戏任务；属于游戏复现或现代实现的缺口才进入任务看板。
- `governance-log.md` 记录脚手架规则变化；问题治理协议和问题文档结构的变化必须写入治理日志。

## 模板

```md
# PG-000 问题名称

状态：待确认。

来源：问题来源。

## 1. 问题定义

问题本体、系统性影响和不治理的后果。

## 2. 证据

- 可复查证据。

## 3. 解决方案

方案、边界和明确不做的事项。

## 4. 测试方案

- 自动检查、搜索验证或人工验收。

## 5. 测试结果

当前结果和后续样本。

## 6. 关闭标准

- 完整关闭条件。
- 只能算部分缓解的情况。
```
