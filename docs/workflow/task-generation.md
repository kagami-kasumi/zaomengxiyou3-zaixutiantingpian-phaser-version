# 任务生成规范

本文定义如何在完整功能条线内生成、拆分和调度 Goal/task。功能条线状态以 `docs/tasks/feature-lines.md` 为准，单次 `/goal` 边界以 `docs/tasks/goal-board.md` 为准，未完成 task 在 `task-board.md`，完成历史在 `task-history.md`。

## 核心模型

本项目使用五个不同层级：

- **机制 `M-*`**：原版游戏事实单元。
- **纵向切片 `VS-*`**：证明一段玩法或技术路径成立的验证样本。
- **功能条线 `LINE-*`**：对用户作出的完整玩家系统交付承诺。
- **Goal 包 `GOAL-*`**：一次 `/goal` 的有界执行与交接单元，最多承受一次 compact。
- **task**：功能条线内部小而可验收的执行单位。

切片、task 或 Goal 完成都不能推出功能条线完成。Goal 完成后当次 `/goal` 结束，但其所属 `LINE-*` 在完整关闭合同满足前继续保持目标所有权。

## Goal 包标准

- 只要存在未完成功能线，`goal-board.md` 必须且只能有一个 `Active` Goal，并且属于唯一 `Active` 功能线。
- Goal 默认只绑定一个 task；最多两个的例外必须有共用产物和验证批次的书面理由。
- Goal 必须有独立交付边界、验收证据、剩余风险和同线下一 Goal，并预计在零次 compact 后完成。
- 每个未完成 task 必须填写 `规模预算`：主工作包最多 2 个、预计上下文压缩为 0、独立验收批次最多 2 个，并填写可操作的 `拆分触发`。
- 同一关卡、同一页面或同一功能线不是合并理由；新资料族/资源派生、多个运行时 owner、端到端运行校准中任意三类同时出现时，必须拆成连续 Goal。
- 同时要求新资料族逆向、大范围实现和端到端验收的工作必须拆成多个 Goal。
- Goal 完成后激活同线下一 Goal，但必须结束当次 `/goal`；禁止隐式续跑。
- `goal-board.md` 只保存未完成 Goal；Goal 完成后直接移除，完成事实和产物随 task 进入 `task-history.md`，不保留 `Done` 行、最近完成摘要或冗余的“下一 Goal”字段。

## 严格单线 WIP=1

- 只要存在未完成功能线，`feature-lines.md` 必须且只能有一条 `Active`。
- 所有 `Ready`、`Blocked` 和当前推荐 task 必须属于这条 `Active` 功能线。
- 非激活功能线的候选 task 只能是 `Planned`。
- 当前 task 完成后，必须生成或选择同线下一 task；不能按全局优先级切换系统。
- 遇到资源、机制、权限、人工验收或工程阻塞时，生成同线解除阻塞 task；阻塞期间不得推进其他功能线。
- 只有激活线满足完整关闭合同后，才允许将下一条 `Planned` 线激活。

## 功能条线标准化

新建功能线前必须由用户确认或根据用户明确目标记录：

- 完整玩家交付范围。
- 权威内容或数据全集。
- 真 UI 和视觉资源要求。
- 正式游戏流程的入口和退出。
- 跨系统集成边界。
- 自动测试、人工试玩和视觉验收要求。
- 明确排除项。
- 覆盖台账与关闭证据落点。

新增功能线写入 `feature-lines.md`；复杂覆盖矩阵放入 `docs/tasks/feature-line-coverage/LINE-*.md`。存在 `Active` 线时，新线只能登记为 `Planned`，不能抢占 WIP。

## task 来源

新 task 必须首先服务当前激活功能线，并来自以下至少一个可复查缺口：

- 功能线覆盖矩阵中的未覆盖项。
- `mechanics-index.md` 中与当前线相关的机制缺口。
- `vertical-slices.md` 中与当前线相关的验证缺口。
- 当前线所需的现代工程或资源基础缺口。
- 当前线的明确阻塞。

不能关联当前 `LINE-*` 的新工作不得进入 `Ready`。AI 工作流、文档职责和任务体系治理不进入游戏看板，只记录到 `docs/workflow/`。

## task 标准

每个 task 必须包含：

- `任务类型`：`TASK-SETTINGS`、`TASK-ARCH` 或 `TASK-SLICE`。
- `功能条线`：唯一所属 `LINE-*` 及其状态。
- `Goal 包`：所属 `GOAL-*` 及其状态；仅 `Split` 父 task 可记为无。
- `目标机制/切片`：至少一个 `M-*` 或 `VS-*`。
- `输入资料`、`输出产物`、`完成定义`、`验收标准`、`禁止范围`、`状态更新`、`推荐后续任务`。

task 的完成定义只判断该工作单元，不得包含未经覆盖证明的整条线关闭结论。推荐后续任务必须属于同一功能线；如果尚需本 task 产出才能编号，应明确写成“依据结果生成同线下一 task”。

每个 task 定义还必须包含：

```md
规模预算：
- 主工作包：1
- 预计上下文压缩：0
- 独立验收批次：1

拆分触发：
- 一旦发现需要新增第二个以上未声明 owner、读取新的资料族或增加独立运行旅程，立即在执行前拆成同线下一 Goal。
```

## ID 与状态

任务 ID：

- `TASK-SETTINGS-###`：行为、机制、内容或资源盘点/逆向。
- `TASK-ARCH-###`：现代工程架构和基础设施。
- `TASK-SLICE-###`：可玩或可验证切片。
- 大任务拆分使用 `004A`、`004B` 等字母后缀，不复用历史 ID。

任务状态：

- `Ready`：属于唯一激活线且可立即执行。
- `Blocked`：属于唯一激活线但有明确阻塞；下一步必须解除本线阻塞。
- `Planned`：非激活线候选，或尚未轮到的同线 task。
- `Split`：已拆成同线子任务，不直接执行。
- `Done`：task 完成并移入历史；不自动改变功能线状态。

## 连续任务生成流程

1. 读取 `feature-lines.md` 和 `goal-board.md`，确认唯一 `Active` 线、唯一 `Active` Goal 及覆盖台账。
2. 读取 `task-board.md`，只执行当前 `Active` Goal 绑定的推荐 task。
3. 从覆盖差异、机制、切片或阻塞中选择同线最小可验收缺口。
4. 生成 task，填写 `LINE-*`、`M-*` / `VS-*` 和完整定义。
5. 同一时刻只保留一个当前推荐；其他同线未来项可 `Planned`。
6. 执行前核对规模预算；超限时先把当前 task 标为 `Split`，生成同线子 task/Goal 并停止实质执行。
7. task 完成后移入历史，更新覆盖台账、Goal 和功能线进度。
8. Goal 绑定 task 全部完成后，将 Goal 从 `goal-board.md` 移除；功能线未关闭时激活或生成同线后续 Goal，然后停止当次 `/goal` 交接。
9. 若遇阻塞，登记阻塞并生成同线解除任务，不得切线。
10. 只有关闭合同全部满足时才把功能线标记 `Done`，随后激活下一条线。

## 逆向 task 生成门禁

生成或拆分 `TASK-SETTINGS-*` 逆向任务时，必须引用 `docs/workflow/reverse-engineering-protocol.md`，并在完成定义和验收标准中覆盖：

- 待证明的可观察问题，而不是笼统的“阅读 AS3”；
- 关卡/对象局部证据及其共享输入、物理、镜头、状态机等实际消费者；
- 视觉/空间行为适用的 SWF 时间轴、注册点、碰撞盒、矩阵和坐标空间；
- 证据矩阵、证据分级、未知项和反证条件；
- 与现代架构无关的行为合同，以及确定性测试和运行观察的双重验证计划。

如果当前 task 只能定位文件或提出推断，完成状态应写成“已定位/待复核”，并生成同线后续逆向 task；不得越级输出“权威实现输入”或把实现 task 设为无阻塞。

## UI 原生化 task 门禁

任何目标、输出或验收涉及 UI、HUD、菜单、页面、按钮、overlay、原版布局或真视觉资源的 task，定义中必须包含独立的 `UI 原生化合同：`，至少写明：

- **显示列表清单**：根/子 Symbol、depth、父子关系、注册点、嵌套矩阵、文字字段、按钮状态、动态 child 和命中区的证据落点；
- **原版视觉基准**：可追溯 Flash 截图/录屏或原 SWF 渲染的版本、入口、状态、舞台尺寸与裁切；缺失时必须阻塞视觉闭合；
- **允许的现代视觉例外**：逐项列出证据缺口、最小边界和用户批准状态；空清单表示禁止新增可见替代层；
- **逐状态验收**：normal/hover/pressed/selected、分页、动态内容、P1/P2、进入/返回中适用的状态；
- **差异证据**：并排/叠图、像素或边缘差异、可见对象差异清单，以及容差解释。

UI 逆向与 UI 实现默认拆成两个 task/Goal。证据 task 未闭合显示列表和原版基准时，实现 task只能保持 `Planned` 或 `Blocked`。仅加载整页真背景、通过路由测试、控制台无错误或业务事务通过，均不能作为 UI 原生化完成证据。

定义中使用以下固定字段，便于工作流校验：

```md
UI 原生化合同：
- 显示列表清单：
- 原版视觉基准：
- 允许的现代视觉例外：
- 逐状态验收：
- 差异证据：
```

## 功能线关闭流程

关闭前必须核对：

- 覆盖台账无未解释缺口。
- 没有同线未完成 task。
- 真 UI、权威内容全集、正式流程、跨系统集成和验证证据满足合同。
- 所有排除项由用户确认。
- `feature-lines.md` 填写具体关闭证据。

局部切片、代表性资源族、系统测试通过或“当前看板无 task”都不能单独作为关闭证据。

## task 模板

```md
### TASK-TYPE-###

任务类型：
- `TASK-SETTINGS` / `TASK-ARCH` / `TASK-SLICE`

功能条线：
- `LINE-*`（必须说明 Active / Planned）

Goal 包：
- `GOAL-*`（必须说明 Active / Planned）

目标机制/切片：
- `M-*` 或 `VS-*`

规模预算：
- 主工作包：1
- 预计上下文压缩：0
- 独立验收批次：1

拆分触发：
- `发现哪些新增范围时立即拆分`

输入资料：
- `必须读取的文档或路径`

输出产物：
- `需要新增或更新的文件`

完成定义：
- `本 task 完成条件，不越级宣告功能线完成`

验收标准：
- `自动检查、搜索或人工验收`

禁止范围：
- `不得切线；本 task 明确不做的事项`

状态更新：
- `feature-lines.md`、覆盖台账、task-board/history 及相关事实表

推荐后续任务：
- `同一 LINE-* 的下一 task`
```

## 创建任务 Prompt 模板

```text
请按 docs/workflow/task-generation.md 为当前唯一 Active 功能线生成分段 Goal 和 task。

要求：
- 先读 docs/tasks/feature-lines.md 和当前线覆盖台账。
- 读 docs/tasks/goal-board.md，每个 Goal 默认仅一 task、预计零次 compact，并核对规模预算与拆分触发。
- task 必须关联当前 LINE-* 和至少一个 M-* / VS-*。
- 只推荐同线下一 task；遇到阻塞就生成同线解除任务。
- 不激活或推进其他功能线。
- 更新 task-board、功能线台账和相关事实文档，不写无关代码。
```
