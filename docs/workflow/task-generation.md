# 任务生成规范

本文定义如何从机制、切片或工程缺口生成标准游戏任务。它用于“新开一个对话专门创建游戏任务”的场景。

执行任务的协议在 `AGENTS.md`；未完成任务状态在 `task-board.md`；已完成任务历史在 `task-history.md`；机制状态在 `mechanics-index.md`。

## 核心原则

任务不是凭感觉产生的。新任务必须来自以下来源之一：

- 机制缺口：`mechanics-index.md` 中有机制未扒清、未复现或状态不可靠。
- 切片缺口：`vertical-slices.md` 中某个切片缺少前置机制或实现步骤。
- 工程基础缺口：现代工程需要一个明确基础设施任务，例如输入、资源、实体管理。

如果一个任务不能对应上述来源之一，不应加入 `task-board.md`。

`task-board.md` 只保存未完成任务。任务完成后应移入 `docs/tasks/task-history.md`，新对话默认不读取历史。

AI 工作流、文档职责、任务体系治理等脚手架维护，不进入游戏任务看板，应记录到 `docs/workflow/governance-log.md`。

## 机制标准化

机制是游戏事实单元。机制可以很小，例如“方向键归玩家 2”；也可以是一个系统入口，例如“怪物生命周期”。

机制在 `mechanics-index.md` 中至少要能回答：

- 机制名称是什么。
- 逆向状态是什么。
- 复现状态是什么。
- 证据或逆向文档在哪里。
- 现代产物在哪里。
- 下一步要做什么。

新增或细化机制时，优先使用表格行；如果机制太复杂，再建立独立逆向文档，并在表格中链接。

机制拆分规则：

- 一个机制如果需要多个任务才能扒清，拆成多个更小机制或多个任务。
- 一个任务可以覆盖多个机制，但必须在任务定义中列出。
- 一个切片通常依赖多个机制，不能把切片本身当作机制。

## 任务标准化

任务是工作单元。一个标准任务必须能让新对话直接执行。

每个任务必须包含：

- `任务类型`：`TASK-SETTINGS`、`TASK-ARCH`、`TASK-SLICE`。
- `目标机制/切片`：这个任务服务哪些机制或切片。
- `输入资料`：必须读取哪些文档、源码路径或现代代码。
- `输出产物`：要新增或更新哪些文件。
- `完成定义`：做完后必须满足哪些条件。
- `验收标准`：如何判断任务完成，而不是只“读过代码”。
- `禁止范围`：本任务明确不做什么。
- `状态更新`：完成后要同步更新哪些表。
- `推荐后续任务`：完成后建议哪个 task 进入 Ready。

## 任务 ID 规则

推荐格式：

- `TASK-SETTINGS-###`：玩法、机制、AS3 逆向。
- `TASK-ARCH-###`：现代工程架构和基础设施。
- `TASK-SLICE-###`：可玩的纵向切片。

拆分任务使用字母后缀：

- `TASK-SETTINGS-004`：大任务，状态为 `Split`。
- `TASK-SETTINGS-004A`、`004B`、`004C`：可执行子任务。

不要复用已完成任务 ID。

## 引用规则

- 机制引用必须使用 `mechanics-index.md` 中的 `M-*` ID。
- 切片引用必须使用 `vertical-slices.md` 中的 `VS-*` ID。
- 任务看板的“目标机制/切片”列至少包含一个 `M-*` 或 `VS-*`。
- 中文机制名可以跟在 ID 后辅助阅读，但不能代替 ID。

## 状态判定

- `Ready`：输入资料明确，依赖满足，一个对话有希望完成。
- `Blocked`：缺前置任务、缺机制事实或缺用户材料。
- `Planned`：方向明确，但当前不是优先级。
- `Split`：任务太大，不能直接执行。
- `Done`：完成定义已满足，相关文档已更新；该任务应从 `task-board.md` 移到 `task-history.md`。

一个任务如果无法写出清晰完成定义，通常不应是 `Ready`。

## 新建任务流程

一个“创建任务”的 prompt 应按这个流程执行：

1. 读取 `AGENTS.md`、`TASK_OUTLINE.md`、`task-board.md`、`mechanics-index.md`、`vertical-slices.md`。
2. 找到机制、切片或工程缺口，并确认对应 `M-*` / `VS-*` ID。
3. 判断是否需要新增机制行或更新已有机制状态。
4. 生成一个或多个任务，并为每个任务写完整标准定义。
5. 把任务写入 `task-board.md`。
6. 如果是由大任务拆分而来，把大任务标为 `Split`。
7. 更新 `task-board.md` 的“当前推荐”。
8. 如果新增机制或改变机制状态，同步更新 `mechanics-index.md`。

## 任务模板

```md
### TASK-TYPE-###

任务类型：

- `TASK-SETTINGS` / `TASK-ARCH` / `TASK-SLICE`

目标机制/切片：

- `M-*` 或 `VS-*`

输入资料：

- `必须读取的文档或路径`

输出产物：

- `需要新增或更新的文件`

完成定义：

- `完成后必须满足的条件`

验收标准：

- `如何判断该任务确实完成`

禁止范围：

- `本任务明确不做的事情`

状态更新：

- `task-board.md`
- `task-history.md`
- `mechanics-index.md`
- `vertical-slices.md`

推荐后续任务：

- `TASK-...`
```

## 创建任务 Prompt 模板

```text
请按 docs/workflow/task-generation.md 创建游戏任务。

目标：围绕【机制/切片/工程缺口】生成一个或多个可执行游戏 task。

要求：
- 先检查 mechanics-index.md 和 vertical-slices.md。
- 每个 task 必须有目标机制/切片、输入资料、输出产物、完成定义、验收标准、禁止范围和推荐后续任务。
- 更新 docs/tasks/task-board.md。
- 如新增或调整机制，同步更新 docs/reverse-engineering/mechanics-index.md。
- 不写游戏代码。
```
