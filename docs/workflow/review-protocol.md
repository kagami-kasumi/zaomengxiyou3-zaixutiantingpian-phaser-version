# Review Protocol

本文规定工程评审的通用流程和输出格式，供 Codex、Claude、DeepSeek 或人工 reviewer 复用。它属于 AI 协作脚手架，不记录具体游戏任务状态。

## 适用范围

使用本协议的场景：

- 用户要求评审代码、任务执行结果、阶段性工程状态或某份评审文档。
- 正式游戏 task 收尾前需要判断实现质量、测试缺口或后续风险。
- 多个 agent 需要对同一批代码用统一标准给出可比较结论。

不使用本协议的场景：

- 只回答一个轻量问题、查配置、解释一段代码或修 typo。
- 只做玩法逆向事实摘录，且不评价现代实现质量。
- 用户明确要求快速主观建议，不要求正式评审。

## 评审原则

- 先看行为正确性、数据一致性、测试缺口和维护风险，再看风格和命名。
- 区分“AS3 已确认事实”“现代实现建议”和“评审者推测”，不要把推测写成原版事实。
- 不要求一次性大重构；所有整改建议应能拆成小 task 或局部改动。
- 评审必须给出可执行结论。只说“代码复杂”“建议优化”不合格。
- 评审不得直接修改 `extracted_flash/`，也不得要求重新生成原始提取结果。
- 对 `src/` 膨胀、重复和 scene 耦合问题，结合 `docs/workflow/code-quality-gates.md` 和 `npm run check:structure` 判断。

## 必读资料

正式评审前按需读取最小集合：

- 入口规则：`AGENTS.md`、`TASK_OUTLINE.md`。
- 代码质量：`docs/workflow/code-quality-gates.md`。
- 涉及 `src/` 边界：`docs/architecture/src-boundaries.md`。
- 涉及具体 task：`docs/workflow/agent-protocol.md`、`docs/tasks/task-board.md`、`docs/reverse-engineering/mechanics-index.md`、`docs/tasks/vertical-slices.md`。
- 涉及已有评审：对应 `docs/评审/*.md`。
- 涉及历史决策：只有需要追溯时读取 `docs/tasks/task-history.md`。

读取大型 Markdown、AS3 或历史文档时，仍遵守入口文档的 UTF-8 和片段读取规则。

## 评审流程

1. 明确评审对象：列出被评审的文件、task、提交范围或阶段范围。
2. 明确评审基准：说明本次按哪些文档、机制事实、测试命令或结构门禁判断。
3. 看行为：检查玩法流程、数值、状态回滚、生命周期、输入、碰撞、存档和 UI 是否有明显风险。
4. 看验证：确认系统测试、构建、结构检查和人工验证分别覆盖了什么，缺了什么。
5. 看结构：检查文件大小、重复代码、scene 耦合、领域规则位置和边界漂移。
6. 看任务状态：如果评审结论影响 task 完成度，同步指出应更新的看板、机制表或切片表。
7. 产出结论：按严重程度列出问题，再给出可执行整改顺序。

## 输出格式

正式评审建议使用以下结构：

```text
评审对象
- 范围：
- 基准：

结论摘要
- 总体判断：
- 是否阻塞继续扩展：

主要问题
- High: ...
- Medium: ...
- Low: ...

文件级发现
- path/to/file.ts:line: 问题、影响、建议。

测试与验证
- 已覆盖：
- 缺口：
- 建议命令：

后续任务建议
- 可直接修复：
- 建议拆 task：

一句话总结
...
```

如果没有发现阻塞问题，也必须明确写出“未发现阻塞问题”，并说明剩余风险或测试缺口。

## 严重程度

- **High**：会导致玩法错误、数据丢失、状态无法恢复、构建/测试失败、任务完成定义不成立，或继续开发会显著放大维护成本。
- **Medium**：不会立刻破坏功能，但已造成明显重复、边界漂移、测试盲区、文件膨胀或后续扩展风险。
- **Low**：局部命名、注释、格式、文档补充或可延后的小型整理。

同一类问题应合并成一个发现，避免用大量重复条目稀释真正风险。

## 评分维度

需要打分时，统一使用 1-10 分，并说明扣分原因：

- 行为一致性：外观、玩法、数值、手感和流程是否贴近原版已确认事实。
- 架构边界：系统、scene、数据模型和 helper 的职责是否清晰。
- 代码质量：重复、命名、类型、生命周期和错误处理是否可维护。
- 测试覆盖：系统测试、构建、结构检查和人工验证是否覆盖主要风险。
- 功能完整度：相对目标 task 或机制的完成定义是否达标。
- 文档同步：task-board、mechanics-index、vertical-slices、history 是否需要同步且已同步。
- 可持续性：继续扩展同一子系统是否会显著增加维护负担。

评分不是必须项。若用户只要求快速 review，可以省略分数，但不能省略关键问题和可执行建议。

## 整改落点

- 小问题：评审中直接列为“可直接修复”。
- 明确游戏复现工作：进入 `docs/tasks/task-board.md`，按 `docs/workflow/task-generation.md` 生成或拆分 task。
- 代码质量门禁或协作流程问题：进入 `docs/workflow/`，并更新 `docs/workflow/governance-log.md`。
- 已完成 task 的事实修正：按历史追溯规则读取并更新 `docs/tasks/task-history.md`。

评审文档可以保存在 `docs/评审/`，但它不能替代任务看板、机制表或切片表的状态更新。
